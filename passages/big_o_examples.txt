! Big O Examples

<img src="assets/big_o_examples.svg" alt="Plot comparing growth rates of O(1), O(log N), O(N), O(N log N), O(N squared), and O(2^N) with ML algorithm examples" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you are organizing a library. With 100 books, any organization system works — you can scan every shelf in a few minutes, toss them into piles by topic, or just remember where each one sits. With a million books, the difference between scanning every shelf (linear) and using a card catalog (logarithmic) is the difference between minutes and days. ''Big O notation'' tells you which algorithms will scale gracefully and which will hit a wall — and in machine learning, where datasets have millions of samples and models have billions of parameters, that wall arrives fast. The constant overhead that seemed negligible at small scale becomes the dominant cost, and the choice of algorithm becomes the choice between a model that trains overnight and one that never finishes.

---

!! The Core Insight

Big O describes how an algorithm's time or memory grows as input size $N$ increases. It ignores constants and lower-order terms — what matters is the //shape// of growth. $O(N)$ means doubling $N$ doubles the work. $O(N^2)$ means doubling $N$ quadruples the work. $O(\log N)$ means doubling $N$ adds only a constant amount of work. In ML, the most important complexity boundaries are: $O(N^2)$ for self-attention (why context length is expensive), $O(N \log N)$ for FFT (why spectral methods are fast), and $O(d^3)$ for matrix inversion (why covariance matrices are expensive). The complexity class of your inner loop determines whether your system works at scale or collapses under its own weight.

See also: [[Big O Notation]], [[Flash Attention]], [[FFT]], [[Transformers]], [[Mamba]].

!! The Complexity Zoo: Concrete ML Examples

''O(1) — Constant:'' Hash table lookup. Checking if a token is in a vocabulary of known size. Evaluating a single neuron with a fixed number of weights. Looking up an embedding by index. Time does not grow with dataset size — a billion-entry hash table is as fast per lookup as a ten-entry one.

''O(log N) — Logarithmic:'' Binary search in a sorted list. Tree traversal in a balanced BST. KD-tree nearest neighbor in low dimensions. Each step halves the search space — doubling $N$ adds exactly one extra step. Searching a sorted list of a billion items takes only about 30 comparisons.

''O(N) — Linear:'' Forward pass through a single layer ($N$ inputs, fixed weights). Computing the mean of $N$ numbers. One epoch of SGD over $N$ training examples. Scanning a dataset once. [[Mamba]]'s sequence processing is $O(L)$ in sequence length $L$ — this is why it scales to long contexts where attention cannot.

''O(N log N) — Linearithmic:'' [[FFT]] (Fast Fourier Transform) — decompose $N$ samples into frequencies. Sorting $N$ items (mergesort, quicksort average). Building a KD-tree. This is the "sweet spot" — almost as fast as linear but far more powerful. Many of the fastest algorithms in signal processing live here.

''O(N^2) — Quadratic:'' Self-attention in [[Transformers]]: every token attends to every other token. Computing a full covariance matrix from $N$ samples. Pairwise distance computation for $N$ points. Naive k-nearest neighbors. This is why transformers struggle with long sequences: 4K context = 16M operations, 128K context = 16B operations — a 1000x increase for a 32x context extension.

''O(N^3) — Cubic:'' Matrix inversion, eigendecomposition, and [[SVD]] of an $N \times N$ matrix. [[Gaussian Processes]] inference (inverting the kernel matrix). Exact kernel methods on $N$ training points. This is why GPs do not scale beyond roughly 10K points without approximations like inducing points.

''O(2^N) — Exponential:'' Brute-force search over all subsets (feature selection with $N$ features). Exact inference in general graphical models. Solving NP-hard problems exactly. Quickly becomes infeasible — $2^{30} \approx 1$ billion, $2^{50} \approx 10^{15}$. No amount of hardware will save you here.

---

!! What the Math Is Actually Doing

Formally: $f(N) = O(g(N))$ means there exist constants $c$ and $N_0$ such that $f(N) \leq c \cdot g(N)$ for all $N > N_0$. It is an //upper bound// on growth rate, describing asymptotic worst-case behavior. Related notations: $\Omega(g)$ = lower bound (the algorithm takes //at least// this long), $\Theta(g)$ = tight bound (both upper and lower match).

''Amortized complexity:'' Some operations are expensive occasionally but cheap on average. Dynamic array resizing costs $O(N)$ when it triggers but $O(1)$ amortized per insertion. In ML, lazy batch processing of a replay buffer has similar amortized characteristics.

''Space complexity:'' Memory growth follows the same notation. Attention is $O(N^2)$ time //and// $O(N^2)$ memory — [[Flash Attention]] reduces memory to $O(N)$ while keeping $O(N^2)$ computation, by never materializing the full attention matrix.

Common ML complexity table:

$$\text{Linear regression (closed form): } O(Nd^2 + d^3)$$
$$\text{SGD (T epochs): } O(TNd)$$
$$\text{Self-attention forward: } O(L^2 d)$$
$$\text{Flash Attention: } O(L^2 d) \text{ time, } O(L) \text{ memory}$$
$$\text{Mamba forward: } O(Ld) \text{ — linear in sequence length}$$
$$\text{k-NN prediction: } O(Nd) \text{ brute force, } O(d \log N) \text{ with trees}$$

---

!! Complexity Tradeoffs in Practice

''Time-space tradeoffs:'' [[Gradient Checkpointing]] trades $O(\sqrt{N})$ memory for roughly 33% more compute — recomputing activations during the backward pass instead of storing them. Caching (memoization) trades memory for time. These tradeoffs are everywhere in ML systems engineering.

''Approximate methods:'' Approximate nearest neighbors (HNSW, LSH) trade accuracy for speed — $O(d \log N)$ instead of $O(Nd)$. [[Quantization]] reduces memory by 2-4x with small accuracy loss. Low-rank approximations turn $O(N^3)$ kernel methods into $O(Nk^2)$ where $k \ll N$.

''Batching and parallelism:'' GPU parallelism means $O(N)$ operations on $N$ items can be effectively $O(1)$ wall-clock time if $N$ fits in a single batch. This is why GPU programming changes the practical calculus of algorithm selection — serial complexity and parallel complexity are different quantities.

''Constants matter:'' Asymptotic analysis hides constants. An $O(N \log N)$ algorithm with a constant of 1000 is slower than an $O(N^2)$ algorithm with a constant of 1 for all $N < 1000$. In practice, profile your actual workload rather than relying purely on Big O.

---

!! When Big O Analysis Breaks Down

''Constants and hardware matter:'' An $O(N \log N)$ FFT with a constant of 1000 is slower than $O(N^2)$ for $N < 1000$. GPU parallelism makes "serial steps" the true bottleneck, not total FLOPs. A highly parallelizable $O(N^2)$ algorithm can beat a serial $O(N \log N)$ on a GPU. Wall-clock time and asymptotic complexity are different things.

''Memory hierarchy dominates:'' Cache-friendly $O(N^2)$ can beat cache-unfriendly $O(N \log N)$ — this is literally why [[Flash Attention]] exists. The algorithm does the same number of FLOPs as standard attention but rearranges memory access patterns to exploit SRAM locality. Modern hardware has 3-4 memory tiers with 10x bandwidth differences between each tier, and Big O tells you nothing about which tier your algorithm lives in.

''Problem-dependent structure:'' Worst-case $O(N^2)$ may be $O(N)$ on typical inputs. Sparse attention exploits the fact that most attention weights are near zero. Early stopping in search prunes the space. Average-case and worst-case can differ by orders of magnitude, and Big O traditionally reports the worst case.

---

!! Technical Details

''Foundational references:'' Knuth (1976) formalized Big O notation for computer science. Cormen, Leiserson, Rivest, and Stein (//Introduction to Algorithms//, known as CLRS) is the standard textbook treatment.

''Average-case vs worst-case:'' Quicksort is $O(N^2)$ worst-case but $O(N \log N)$ expected with random pivots. In ML, stochastic gradient descent has expected convergence rates that differ from worst-case bounds.

''Amortized analysis:'' The potential method and the accounting method formalize amortized bounds. Hash table operations are $O(1)$ amortized despite occasional $O(N)$ rehashing.

''Parameterized complexity:'' Fixed-parameter tractable (FPT) algorithms are exponential in a parameter $k$ but polynomial in $N$: $O(2^k \cdot N)$. Relevant for problems like feature selection where the number of selected features $k$ is small.

''Training vs inference:'' In ML, the distinction between training complexity and inference complexity is critical. Training a transformer is $O(TLN^2d)$ over $T$ epochs, but a single inference pass is $O(L^2 d)$. Training is done once (or rarely); inference runs continuously. Optimizing inference complexity often matters more in deployment.

''Space-time product:'' Some theoretical results bound the product of time and space. [[Flash Attention]] achieves optimal time-space product for attention: $O(N^2 d)$ time $\times$ $O(N)$ space = $O(N^3 d)$, which is provably optimal for exact attention.
