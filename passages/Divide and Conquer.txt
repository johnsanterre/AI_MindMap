! Divide and Conquer

<img src="assets/divide_and_conquer.svg" alt="Diagram showing divide and conquer strategy: a problem of size n splits into subproblems of size n/2, each solved recursively, then combined into a final solution, with recurrence T(n) = aT(n/b) + f(n)" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you need to alphabetize a thousand library cards. Doing it all at once is overwhelming — you would need to compare every card against every other card. But what if you split the pile in half, alphabetized each half separately, and then merged the two sorted halves together? Each half-pile is easier, and merging two already-sorted stacks is straightforward — just compare the top cards and pick the smaller one. That is ''divide and conquer'' in a nutshell: break a hard problem into smaller versions of itself, solve those smaller versions, and combine the results. It is one of the most powerful and universal strategies in all of algorithm design.

!! The Core Insight

Divide and conquer rests on a single observation: many problems have ''self-similar structure''. A sorting problem on $n$ elements looks exactly like a sorting problem on $n/2$ elements — just smaller. A search problem on a sorted array of $n$ elements can be reduced to a search on $n/2$ elements by checking the middle. The strategy exploits this self-similarity through three steps:

''1. Divide:'' Split the problem into $a$ smaller subproblems, each roughly $1/b$ the size of the original.

''2. Conquer:'' Solve each subproblem recursively. When a subproblem is small enough (the //base case//), solve it directly.

''3. Combine:'' Merge the subproblem solutions into a solution for the original problem. The cleverness of the algorithm usually lives here — the combine step determines the overall efficiency.

The magic is that dividing a problem in half and solving both halves //plus// combining is often dramatically cheaper than tackling the original directly. Sorting $n$ items by brute force takes $O(n^2)$ comparisons, but merge sort — which divides, conquers, and combines — does it in $O(n \log n)$.

!! Classic Divide and Conquer Algorithms

''Merge Sort'' divides an array in half, recursively sorts each half, and merges the two sorted halves in $O(n)$ time. Total cost: $T(n) = 2T(n/2) + O(n) = O(n \log n)$. It is the textbook example and one of the most elegant [[Sorting Algorithms]] ever designed.

''Binary Search'' divides a sorted array in half and discards the irrelevant half. Only one subproblem survives: $T(n) = T(n/2) + O(1) = O(\log n)$. This is divide and conquer at its most efficient — eliminating half the search space at every step.

''Quicksort'' picks a //pivot//, partitions the array into elements smaller and larger than the pivot, and recurses on both partitions. Expected time is $O(n \log n)$, but worst-case is $O(n^2)$ if the pivot is consistently bad (e.g., always the smallest element). Randomized pivot selection makes the worst case extremely unlikely.

''Strassen's Matrix Multiplication'' multiplies two $n \times n$ matrices using 7 recursive multiplications of $n/2 \times n/2$ matrices instead of the naive 8. This gives $T(n) = 7T(n/2) + O(n^2) = O(n^{2.807})$, beating the naive $O(n^3)$.

''Karatsuba Multiplication'' multiplies two $n$-digit numbers using 3 multiplications of $n/2$-digit numbers instead of 4. Cost: $T(n) = 3T(n/2) + O(n) = O(n^{1.585})$, faster than the schoolbook $O(n^2)$.

''Closest Pair of Points'' finds the two nearest points in a plane. Divide the points by a vertical line, solve each half, then check pairs that straddle the boundary. The combine step is the subtle part — only a narrow strip needs checking, and a clever argument bounds comparisons to $O(n)$, giving $O(n \log n)$ overall.

!! What the Math Is Actually Doing

The runtime of a divide and conquer algorithm is captured by a ''recurrence relation'':

$$T(n) = aT\!\left(\frac{n}{b}\right) + f(n)$$

where $a$ is the number of subproblems, $b$ is the factor by which the problem shrinks, and $f(n)$ is the cost of dividing and combining. The ''Master Theorem'' (see [[Recurrence Relations]]) solves this recurrence in three cases by comparing $f(n)$ to $n^{\log_b a}$:

''Case 1:'' If $f(n) = O(n^{\log_b a - \epsilon})$ for some $\epsilon > 0$, then $T(n) = \Theta(n^{\log_b a})$. The work is dominated by the leaves of the recursion tree.

''Case 2:'' If $f(n) = \Theta(n^{\log_b a})$, then $T(n) = \Theta(n^{\log_b a} \log n)$. The work is evenly spread across all levels of the recursion tree.

''Case 3:'' If $f(n) = \Omega(n^{\log_b a + \epsilon})$ and $af(n/b) \leq cf(n)$ for some $c < 1$, then $T(n) = \Theta(f(n))$. The combine step dominates.

For merge sort: $a = 2$, $b = 2$, $f(n) = O(n)$. Since $n^{\log_2 2} = n^1 = n = f(n)$, we are in Case 2: $T(n) = \Theta(n \log n)$.

For Strassen: $a = 7$, $b = 2$, $f(n) = O(n^2)$. Since $n^{\log_2 7} \approx n^{2.807} \gg n^2$, we are in Case 1: $T(n) = \Theta(n^{\log_2 7})$.

The recursion tree visualization makes this intuitive. At the top level, you do $f(n)$ work. At the next level, you do $a \cdot f(n/b)$ work across all subproblems. The tree has $\log_b n$ levels, and the total work is the sum across all levels. Whether the total is dominated by the root, the leaves, or evenly balanced determines which Master Theorem case applies.

!! Divide and Conquer vs Other Paradigms

Divide and conquer splits a problem into //independent// subproblems. [[Greedy Algorithms]] make a single locally optimal choice at each step without looking back. ''Dynamic programming'' solves //overlapping// subproblems — when the same subproblem appears multiple times, dynamic programming caches results to avoid redundant work, while pure divide and conquer would recompute them.

A classic example of the distinction: computing the $n$-th Fibonacci number. A naive divide and conquer approach ($F(n) = F(n-1) + F(n-2)$) gives exponential time $O(2^n)$ because the same subproblems are solved repeatedly. Dynamic programming stores each $F(k)$ once, reducing the cost to $O(n)$.

''When to choose divide and conquer:'' Use it when subproblems are independent and roughly equal in size. Use dynamic programming when subproblems overlap. Use greedy algorithms when a local choice provably leads to a global optimum.

!! When Divide and Conquer Breaks Down

''1. Overlapping subproblems.'' When the recursive decomposition produces the same subproblems multiple times, divide and conquer wastes enormous effort recomputing them. The Fibonacci recurrence is the simplest example: $F(5)$ needs $F(4)$ and $F(3)$, but $F(4)$ also needs $F(3)$, and the redundancy grows exponentially. The fix is dynamic programming — memoize subproblem results. Any time you see a divide and conquer solution with exponential runtime, check whether subproblems overlap.

''2. Unbalanced partitions.'' Divide and conquer assumes subproblems are roughly equal in size. When they are not, performance degrades sharply. Quicksort's worst case ($O(n^2)$) occurs when every partition puts all but one element on one side — the recursion tree degenerates from $O(\log n)$ depth to $O(n)$ depth. Median-of-three or randomized pivot selection mitigates this, but any divide and conquer algorithm is vulnerable to pathological inputs that produce lopsided splits.

''3. High combine cost.'' The combine step can dominate and negate the benefits of dividing. If merging $a$ subproblem solutions takes $O(n^2)$ work, dividing the problem may not help — you might as well solve it directly. Algorithms where the combine step is the bottleneck need clever restructuring (like the closest-pair strip argument) to keep the combine cost low.

!! Technical Details

''Master Theorem regularity condition:'' Case 3 requires the //regularity condition// $af(n/b) \leq cf(n)$ for some constant $c < 1$. This ensures $f(n)$ is "polynomially larger" than $n^{\log_b a}$ in a well-behaved way. Most practical combine functions satisfy it.

''Akra-Bazzi theorem:'' Generalizes the Master Theorem to recurrences of the form $T(n) = \sum_i a_i T(n/b_i) + f(n)$ where subproblems can have //different// sizes. It states $T(n) = \Theta\!\left(n^p\!\left(1 + \int_1^n \frac{f(u)}{u^{p+1}}\,du\right)\right)$ where $p$ satisfies $\sum_i a_i / b_i^p = 1$.

''Recursion overhead:'' In practice, divide and conquer algorithms switch to a direct (non-recursive) algorithm when the subproblem size falls below a threshold (e.g., insertion sort for arrays under 16 elements in merge sort implementations). This avoids the overhead of function calls on tiny inputs.

''Parallelism:'' Divide and conquer is naturally parallel — independent subproblems can be solved on separate processors. The ''work-span model'' measures parallel efficiency: total work $T_1$ divided by span $T_\infty$ (longest dependent chain) gives the maximum useful parallelism $T_1/T_\infty$. Merge sort has work $O(n \log n)$ and span $O(\log^2 n)$ with parallel merging, giving parallelism $O(n/\log n)$.

''Notable applications in ML:'' The FFT (Fast Fourier Transform) is divide and conquer — $T(n) = 2T(n/2) + O(n) = O(n \log n)$ — and underlies efficient convolution in CNNs. $k$-d trees for nearest neighbor search recursively partition space, and randomized partition trees enable fast approximate nearest neighbor queries in high-dimensional spaces.

Related topics: [[Recurrence Relations]], [[Sorting Algorithms]], [[Greedy Algorithms]].
