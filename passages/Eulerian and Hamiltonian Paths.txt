! Eulerian and Hamiltonian Paths

<img src="assets/eulerian_hamiltonian_paths.svg" alt="Side-by-side comparison: Eulerian path visiting every edge once, Hamiltonian path visiting every vertex once" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine a postal worker who must drive down every street in a neighborhood exactly once before returning to the post office. Now imagine a traveling salesperson who must visit every city exactly once. Both problems involve traversing a graph, but they ask fundamentally different questions — and that difference spans a chasm in computational complexity. The ''Eulerian path'' asks whether you can walk every //edge// exactly once. The ''Hamiltonian path'' asks whether you can visit every //vertex// exactly once. Euler's version has an elegant, instant test. Hamilton's version is one of the hardest problems in all of computer science.

!! The Core Insight

The deep asymmetry is this: edges are ''local'' structure, vertices are ''global'' structure. Whether you can traverse every edge depends only on vertex degrees — a property you can check by looking at each node independently. Whether you can visit every vertex depends on the //entire topology// of the graph, and no local check suffices. This is why Eulerian paths are solvable in polynomial time while Hamiltonian paths are [[NP-Complete|NP-Completeness]]. Two questions that //sound// almost identical live in completely different complexity universes.

!! K&#246;nigsberg Bridges and Euler's Theorem

In 1736, Leonhard Euler asked whether the citizens of K&#246;nigsberg could walk across all seven bridges of the Pregel River exactly once. He abstracted the problem: landmasses became vertices, bridges became edges. His answer — //no// — launched [[Graph Theory|Graph Theory]].

''Euler's Theorem'' gives a complete characterization:

* A connected graph has an ''Eulerian circuit'' (closed path visiting every edge once) if and only if every vertex has ''even degree''.
* A connected graph has an ''Eulerian path'' (open, start &#8800; end) if and only if exactly two vertices have ''odd degree''. Those two vertices must be the start and end points.

K&#246;nigsberg had four vertices all with odd degree, so no Eulerian path existed. The proof is constructive: Hierholzer's algorithm (1873) finds an Eulerian circuit in $O(|E|)$ time by stitching together cycles greedily. This is //linear// in the number of edges — about as efficient as an algorithm can be.

!! Hamiltonian Paths and the Complexity Barrier

William Rowan Hamilton posed the complementary question in 1857: can you visit every vertex exactly once? Unlike Euler's clean degree criterion, no simple necessary-and-sufficient condition exists for Hamiltonian paths. The problem is ''NP-complete'', meaning:

* Given a claimed solution, you can //verify// it in polynomial time (just check that every vertex appears exactly once).
* No known algorithm can //find// a solution faster than exponential time in the worst case.

Two classical ''sufficient'' conditions provide partial answers:

''Dirac's Theorem (1952):'' If every vertex in a graph with $n \geq 3$ vertices has degree $\geq n/2$, then the graph has a Hamiltonian cycle.

''Ore's Theorem (1960):'' If for every pair of non-adjacent vertices $u, v$ we have $\deg(u) + \deg(v) \geq n$, then the graph has a Hamiltonian cycle. (Dirac's theorem is a special case.)

These are //sufficient but not necessary// — many graphs with lower degrees still have Hamiltonian paths. The gap between sufficient conditions and the general problem is where the NP-completeness lives.

!! What the Math Is Actually Doing

''Eulerian path existence'' reduces to a degree-parity check. Define the degree sequence $d_1, d_2, \ldots, d_n$. Then:

$$\text{Eulerian circuit exists} \iff \forall\, i: d_i \equiv 0 \pmod{2}$$

$$\text{Eulerian path exists} \iff |\{i : d_i \equiv 1 \pmod{2}\}| \in \{0, 2\}$$

Both checks run in $O(n)$ time. Hierholzer's algorithm then constructs the path in $O(|E|)$ by maintaining a stack of unused edges.

For ''Hamiltonian paths'', the best exact algorithm uses [[Dynamic Programming|Dynamic Programming]] over subsets (Bellman-Held-Karp, 1962):

$$\text{dp}[S][v] = \begin{cases} \text{true} & \text{if there is a path visiting exactly the vertices in } S \text{ ending at } v \end{cases}$$

$$\text{dp}[S][v] = \bigvee_{u \in S \setminus \{v\},\, (u,v) \in E} \text{dp}[S \setminus \{v\}][u]$$

This runs in $O(2^n \cdot n^2)$ time and $O(2^n \cdot n)$ space — exponential, but far better than the na&#239;ve $O(n!)$ brute force. For the ''Travelling Salesman Problem'' (TSP), the same DP framework minimizes total edge weight over all Hamiltonian cycles.

!! Connections to ML and Applied CS

''De Bruijn sequences and genomics:'' A de Bruijn sequence of order $k$ on alphabet $\Sigma$ contains every possible $k$-mer as a substring. Finding it reduces to an ''Eulerian path'' in the de Bruijn graph, where nodes are $(k{-}1)$-mers and edges represent $k$-mers. Modern genome assemblers (Velvet, SPAdes) use exactly this construction — DNA sequencing is, at its core, an Euler path problem.

''TSP heuristics in route optimization:'' The Hamiltonian cycle problem generalizes to TSP, which drives logistics, circuit board drilling, and delivery routing. Since exact solutions are intractable for large instances, ML approaches have emerged: pointer networks (Vinyals et al., 2015) use attention-based sequence-to-sequence models to output city orderings; reinforcement learning methods (Bello et al., 2016) train policies that construct tours incrementally. Graph neural networks encode city coordinates and learn edge selection probabilities, approaching within 1-2% of optimal on benchmarks with thousands of cities.

''[[Combinatorics|Combinatorics]] in algorithm design:'' Both problems illustrate how [[Graph Theory|Graph Theory]] structures appear throughout ML — from constructing training data sequences (Eulerian paths through data graphs) to solving assignment and routing problems that arise in distributed training and resource allocation.

!! When Eulerian/Hamiltonian Analysis Breaks Down

''(1) Directed and mixed graphs complicate the degree conditions.'' Euler's theorem for directed graphs requires in-degree = out-degree at every vertex (for a circuit), which is more restrictive. Mixed graphs (some directed, some undirected edges) make even the Eulerian path problem NP-complete. Many real-world networks are directed, so the clean undirected theory does not directly apply.

''(2) Hamiltonian sufficient conditions are far from tight.'' Dirac's and Ore's theorems demand high minimum degree ($\geq n/2$), but real-world graphs are typically sparse. A social network with millions of nodes and average degree 100 fails these conditions massively, yet may well contain Hamiltonian paths. The theorems provide no guidance for sparse graphs, which is where most practical instances live.

''(3) Approximate and heuristic TSP solutions lack worst-case guarantees on general graphs.'' Christofides' algorithm guarantees a 3/2-approximation for metric TSP, but general (non-metric) TSP has no constant-factor approximation unless P = NP. ML-based solvers perform well empirically but can fail catastrophically on adversarial or out-of-distribution instances, and their learned heuristics do not transfer reliably across graph families.

!! Technical Details

''Historical origins:'' Euler (1736) solved the K&#246;nigsberg bridges problem, founding both graph theory and topology. Hamilton (1857) invented the "Icosian game" — finding a Hamiltonian cycle on a dodecahedron — and sold it commercially as a puzzle.

''Hierholzer's algorithm (1873):'' Start at any vertex, follow edges (removing them) until returning to start, forming a cycle. If unused edges remain, pick a vertex on the current cycle with unused edges, form a new cycle, and splice it in. Repeat until all edges used. Time: $O(|E|)$.

''Chinese Postman Problem:'' The optimization version of the Eulerian problem — find the minimum-weight closed walk covering every edge at least once. For undirected graphs, this is solvable in polynomial time by finding a minimum-weight perfect matching on odd-degree vertices and duplicating those edges.

''Held-Karp bound:'' The linear programming relaxation of TSP (minimum spanning 1-tree) provides a lower bound within about 1% of optimal for random Euclidean instances. This bound guides branch-and-bound solvers like Concorde, which has solved instances with 85,900 cities to provable optimality.

''Christofides-Serdyukov algorithm (1976):'' For metric TSP, compute MST, find minimum-weight perfect matching on odd-degree MST vertices, combine, find Eulerian circuit, shortcut repeated vertices. Guarantees tour weight $\leq \frac{3}{2} \cdot \text{OPT}$. This stood as the best approximation ratio for 45 years until Karlin, Klein, and Gharan (2021) achieved $\frac{3}{2} - \epsilon$.

See also: [[Graph Theory|Graph Theory]], [[NP-Completeness|NP-Completeness]], [[Combinatorics|Combinatorics]], [[Dynamic Programming|Dynamic Programming]].
