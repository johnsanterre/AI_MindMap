! Model Merging
<img src="assets/model_merging.svg" alt="Diagram showing three fine-tuned models being merged into a single model through TIES, DARE, SLERP, or task arithmetic" style="max-width:100%;margin:0.5em 0 1em;">

Imagine three chefs who all trained at the same culinary school but then spent years specializing — one in pastry, one in sushi, one in barbecue. Model merging is like combining their specialized knowledge back into a single chef who can do all three, without sending anyone back to school. You take the weight differences each specialist developed and stitch them together into one model, with no additional training required.

!! The Core Insight

When you ''fine-tune'' a pre-trained model on different tasks, each fine-tuned copy drifts away from the base model in a specific direction in weight space. These directions are largely //orthogonal// — the code-generation specialist adjusts different parameters than the math-reasoning specialist. Model merging exploits this by combining the task-specific weight deltas back into a single model.

The surprising finding: simple arithmetic on model weights — averaging, interpolating, or selectively combining them — often produces a model that inherits capabilities from all donors. No gradient updates, no training data, no GPU hours. Just weight manipulation.

Related methods: <<link "Fine-Tuning" "Fine-Tuning">><</link>>, <<link "LoRA" "LoRA">><</link>>, <<link "Transfer Learning" "Transfer Learning">><</link>>, <<link "Ensemble Methods" "Ensemble Methods">><</link>>, <<link "Weight Decay" "Weight Decay">><</link>>.

!! The Merging Toolkit

Four major techniques dominate the merging landscape:

''Task Arithmetic (Ilharco et al., 2023):'' The simplest approach. Compute a //task vector// $\tau = \theta_{\text{ft}} - \theta_{\text{base}}$ for each fine-tuned model, then add scaled task vectors back to the base: $\theta_{\text{merged}} = \theta_{\text{base}} + \lambda_1 \tau_1 + \lambda_2 \tau_2 + \ldots$ The scaling coefficients $\lambda_i$ control how much of each skill to inject.

''TIES-Merging (Yadav et al., 2023):'' Task arithmetic suffers from //sign conflicts// — different task vectors may push the same parameter in opposite directions. TIES (Trim, Elect Sign, and Merge) addresses this in three steps: (1) trim small-magnitude values to zero, (2) resolve sign conflicts by majority vote, (3) average only the values that agree.

''DARE (Yu et al., 2024):'' Most delta parameters are redundant. DARE randomly drops a large fraction (e.g., 90–99%) of delta values, then rescales the survivors by $1/(1-p)$ to preserve the expected magnitude. This sparsification dramatically reduces interference between task vectors.

''SLERP (Shoemake, 1985; adapted for LLMs):'' Instead of linear interpolation between two models, SLERP (Spherical Linear Interpolation) traces a great-circle arc on the weight-space hypersphere, better preserving the norm of the weight vectors. Widely used for merging exactly two models.

!! What the Math Is Actually Doing

The foundation of model merging is the ''task vector''. Given base weights $\theta_{\text{base}}$ and a fine-tuned model $\theta_{\text{ft}}$:
$$\tau = \theta_{\text{ft}} - \theta_{\text{base}}$$

''Task arithmetic'' merges $n$ models by:
$$\theta_{\text{merged}} = \theta_{\text{base}} + \sum_{i=1}^{n} \lambda_i \tau_i$$

''TIES-Merging'' first creates a trimmed mask $M_i$ that zeros out the bottom $k\%$ of $|\tau_i|$ values, then resolves signs via element-wise majority vote across task vectors:
$$\gamma_j = \text{sign}\!\left(\sum_i M_i^{(j)} \tau_i^{(j)}\right)$$
Only entries whose sign matches $\gamma_j$ are averaged, producing a cleaner merged delta.

''DARE'' applies a random binary mask $m \sim \text{Bernoulli}(1-p)$ to each task vector:
$$\tilde{\tau}_i = \frac{m \odot \tau_i}{1-p}$$
This is analogous to ''dropout'' applied to weight deltas rather than activations. The rescaling ensures $\mathbb{E}[\tilde{\tau}_i] = \tau_i$.

''SLERP'' between two weight vectors $\theta_A$ and $\theta_B$ with interpolation parameter $t$:
$$\theta_{\text{merged}} = \frac{\sin((1-t)\Omega)}{\sin \Omega}\,\theta_A + \frac{\sin(t\,\Omega)}{\sin \Omega}\,\theta_B$$
where $\Omega = \arccos\!\left(\frac{\theta_A \cdot \theta_B}{\|\theta_A\|\,\|\theta_B\|}\right)$ is the angle between the weight vectors.

!! Practical Merging Recipes

''Choosing a method:'' for two models, SLERP is often the default. For three or more, TIES or DARE combined with task arithmetic tend to outperform naive averaging. LoRA weights merge particularly well because their low-rank structure constrains the delta space.

''Hyperparameter tuning:'' the scaling coefficients $\lambda_i$ matter enormously. A common approach is to sweep $\lambda$ on a small validation set. Values between 0.3 and 1.0 per task are typical.

''Merging LoRA adapters:'' because <<link "LoRA" "LoRA">><</link>> produces low-rank deltas $\Delta W = BA$, you can merge at the adapter level before folding into the base model. This is computationally cheaper and often cleaner than merging full-rank weights.

''Evolutionary merging (Akiba et al., 2024):'' treat the merging recipe itself — which layers to merge, which method to use per layer, what $\lambda$ values — as a search problem and optimize with evolutionary algorithms. This "mergekit-evolve" approach has produced state-of-the-art open models.

''The MergeKit ecosystem:'' the open-source MergeKit library implements all major merging methods and has become the standard tool. Many top models on the Open LLM Leaderboard are merges, not models trained from scratch.

!! When Model Merging Breaks Down

''Catastrophic interference from conflicting representations:'' when two fine-tuned models learn fundamentally incompatible internal representations — for instance, one model repurposes the same neurons for coding that another uses for poetry — merging produces destructive interference. TIES and DARE mitigate this but cannot eliminate it when the tasks are deeply entangled in weight space.

''Capability dilution with too many donors:'' each additional model added to a merge dilutes the contribution of every other model. Beyond 3-4 donors, merged models often exhibit a "jack of all trades, master of none" degradation where no single capability matches the original specialist. The scaling coefficients become increasingly difficult to balance.

''Base model mismatch:'' merging only works when all models share the same base architecture and (ideally) the same base checkpoint. Even subtle differences — different tokenizers, different pre-training runs, or architectural modifications — make the task vectors incommensurable and the merge meaningless.

!! Technical Details

''Key papers:'' Ilharco et al. 2023 ("Editing Models with Task Arithmetic"); Yadav et al. 2023 ("TIES-Merging: Resolving Interference When Merging Models"); Yu et al. 2024 ("Language Models are Super Mario: Absorbing Abilities from Homologous Models as a Free Lunch" — the DARE paper); Akiba et al. 2024 ("Evolutionary Optimization of Model Merging Recipes").

''Why it works — the linear mode connectivity hypothesis:'' fine-tuned models from the same base tend to lie in the same loss basin, meaning linear interpolation between them stays in a region of low loss. This was formalized by Frankle et al. (2020) and underpins all averaging-based merging.

''Frankenmerging:'' layer-wise merging where different layers come from different models entirely. For example, use Model A's attention layers but Model B's feed-forward layers. This is more art than science but has produced strong results.

''Relationship to ensembles:'' <<link "Ensemble Methods" "Ensemble Methods">><</link>> run multiple models at inference and combine //outputs//. Model merging combines //weights// before inference, producing a single model with no extra compute cost. The trade-off: ensembles preserve each model's full capability; merges compress capabilities but may lose fidelity.

''Merging and RLHF:'' models fine-tuned with reinforcement learning from human feedback often merge poorly because RLHF induces large, structured weight changes that conflict across models. DPO-tuned models tend to merge more cleanly.

''Parameter count:'' merging produces a model of //identical// size to each donor — no parameter increase. This contrasts with mixture-of-experts approaches that scale parameter count.
