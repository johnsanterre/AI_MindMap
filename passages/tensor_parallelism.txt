! Tensor Parallelism
<img src="assets/tensor_parallelism.svg" style="max-width:100%;margin:0.5em 0 1em 0;" alt="Tensor parallelism diagram showing weight matrix sharding across 4 GPUs with all-reduce, plus strategy comparison">

A single person can multiply a 1000×1000 matrix by a vector — slowly. Give half the matrix to a colleague, they each compute their half simultaneously, then you combine results with a quick addition. Tensor parallelism does exactly this for neural networks: split weight matrices across multiple GPUs, compute in parallel, all-reduce the results. The model runs faster, and models too large to fit on a single GPU become trainable.

!! The Core Insight

''Tensor parallelism'' (TP), introduced by Shoeybi et al. 2019 in Megatron-LM, splits individual weight matrices across multiple GPUs. For a linear layer $Y = XW$, split $W$ column-wise across $G$ GPUs: $W = [W_1 \mid W_2 \mid \cdots \mid W_G]$. Each GPU $i$ computes $Y_i = X W_i$ locally. An all-reduce (sum or gather) across GPUs then reconstructs $Y$.

TP is one pillar of ''3D parallelism'', combined with data parallelism and pipeline parallelism, and was used to train 530B+ parameter models. Related concepts: [[Deep Learning]], [[Large Language Models]], [[Hardware Lottery]], [[Transformers]], [[Mixed Precision Training]], [[Gradient Checkpointing]].

!! Column-wise and Row-wise Sharding

A typical Transformer MLP block computes $Y = \text{GeLU}(X W_1) W_2$.

''Column-parallel $W_1$:'' split columns so each GPU produces a different subset of output features. No inter-GPU communication is needed between GeLU and the next weight.

''Row-parallel $W_2$:'' split rows so each GPU receives the corresponding input features and produces a partial sum. An all-reduce then sums the partial outputs.

For attention layers: $Q$, $K$, $V$ projection matrices are split so each GPU handles a different subset of heads. An all-reduce follows the output projection. The key design principle is to minimize synchronization — communication happens only at layer boundaries, not inside them.

!! What the Math Is Actually Doing

Forward pass:
$$X_{\text{global}} \xrightarrow{\text{broadcast}} \text{each GPU} \xrightarrow{Y_i = f(X, W_i)} \text{local compute} \xrightarrow{\text{all-reduce}} Y = \sum_i Y_i$$

Backward pass mirrors this pattern: gradient all-reduces propagate weight updates back to each shard.

Communication volume per all-reduce: $O(\text{batch} \times \text{seq} \times \text{hidden})$ — independent of weight matrix size. The critical metric is the ''compute-to-communication ratio''. On NVLink (600–900 GB/s on A100/H100), TP scales efficiently up to 8 GPUs per node. Across nodes on InfiniBand (400 Gb/s), TP communication becomes a bottleneck and pipeline parallelism is preferred instead.

!! 3D Parallelism

Modern large-scale training combines three orthogonal strategies:

''Data Parallelism (DP):'' replicate the full model on each worker; each processes a different batch; gradient all-reduce synchronizes weights.

''Tensor Parallelism (TP):'' split weight matrices within a node over fast NVLink. No replication of weights — each GPU holds only $1/G$ of each matrix.

''Pipeline Parallelism (PP):'' assign different transformer layers to different nodes. Micro-batches flow through the pipeline; the main cost is pipeline bubble overhead.

''ZeRO'' (Rajbhandari et al. 2020) complements DP by sharding optimizer states, gradients, and parameters across data-parallel workers, reducing per-GPU memory proportionally to the DP degree. Training the Megatron-Turing 530B model used 280 nodes × 8 A100s with TP=8 within each node, PP=35 across nodes, and DP=8 across pipeline replicas.

!! When Tensor Parallelism Breaks Down

''Slow interconnect:'' TP requires a fast all-reduce at every layer boundary. On slow inter-node links (e.g., standard Ethernet), communication latency dominates compute time and throughput collapses. TP is best confined within a single node on NVLink.

''Small models:'' for a model that fits comfortably on one GPU, the overhead of all-reduce synchronization outweighs any speedup from splitting the matrix multiply. TP only pays off when the weight matrix is large relative to communication cost.

''Non-linear bottlenecks:'' operations like LayerNorm and Dropout require access to the full feature vector and cannot be trivially sharded. These must be replicated, handled via sequence parallelism (splitting the sequence dimension instead), or treated as synchronization points — adding complexity and potential inefficiency.

!! Technical Details

Key papers: Shoeybi et al. 2019 (Megatron-LM); Rajbhandari et al. 2020 (ZeRO); Narayanan et al. 2021 (3D parallelism for 1T-parameter models); Smith et al. 2022 (Megatron-Turing NLG 530B).

Hardware: NVLink bandwidth 600 GB/s (A100) / 900 GB/s (H100). InfiniBand HDR: ~200 Gb/s per port; NDR: ~400 Gb/s. PCIe 4.0: ~32 GB/s — too slow for TP.

All-reduce algorithms: ring all-reduce achieves $O(N)$ bandwidth efficiency (independent of GPU count); used by NCCL. Recursive halving-doubling reduces latency for small tensors.

''Sequence parallelism'' (Korthikanti et al. 2022): shard the sequence dimension for LayerNorm and Dropout, complementing TP for attention and MLP sharding.

Flash Attention integrates cleanly with tensor parallelism: attention heads are split across GPUs, and Flash Attention's tiled kernel runs independently per head shard with no inter-GPU communication until the output projection all-reduce.
