! Flash Attention

<img src="assets/flash_attention.svg" alt="Diagram illustrating Flash Attention tiled computation vs standard attention" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine doing a jigsaw puzzle with ten thousand pieces. The standard approach lays every single piece out across an enormous table — you can see everything, but the table is in a cold warehouse a block away, and every time you need a piece you have to run and fetch it. Flash Attention is like keeping a small, fast tray at your elbow: you work through the puzzle in small batches, carrying only the tiles you need right now, and never once dragging the whole pile into the slow warehouse. The puzzle comes out identical either way — but one approach takes hours and the other takes minutes.

---

!! The Core Insight

Flash Attention is an //IO-aware// algorithm: its central insight is that the bottleneck for computing [[scaled dot-product attention]] on modern GPUs is not the number of floating-point operations, but the number of times data must travel between fast on-chip memory (SRAM) and slow off-chip memory (HBM). Standard attention materialises a full $N \times N$ matrix of attention scores in HBM, reads it back for the softmax, writes the softmax output, reads it again for the weighted sum — an avalanche of round-trips that swamps the arithmetic units while they wait.

Flash Attention, introduced by Dao et al. (2022), restructures the [[Attention]] computation so that tiles of the query, key, and value matrices are loaded once into SRAM, all arithmetic for that tile is done in place, and only the final output tile is written back. The $N \times N$ score matrix is //never// fully materialised. This is possible because softmax can be computed incrementally using an online algorithm that tracks a running maximum and a running normalisation constant — merging tiles exactly as though the whole matrix had been available at once.

The practical consequence is that Flash Attention matches the numerical output of standard [[Transformers]] attention to floating-point precision, while reducing memory reads and writes from $O(N^2)$ to $O(N^2 / M)$, where $M$ is the SRAM size. For [[Multi-head Attention]] on long sequences this is the difference between fitting in memory and running out, and between waiting minutes and waiting seconds. This algorithmic efficiency is a direct illustration of the [[Hardware Lottery]]: researchers who understood the hardware won; those who ignored it lost.

---

!! The Memory Hierarchy Problem

A modern GPU like the NVIDIA A100 has three relevant memory tiers. ''Registers'' are the fastest but hold only a handful of values per thread. ''SRAM'' (also called shared memory or L2 cache depending on context) sits on-chip, delivers roughly 19 TB/s bandwidth, and totals around 20 MB on the A100 (50 MB on the H100). ''HBM'' (High Bandwidth Memory) is the large DRAM attached to the GPU — 80 GB on the A100 — but despite the name its bandwidth (~2 TB/s) is ten times slower than SRAM and it has orders-of-magnitude higher latency.

Standard attention allocates the $N \times N$ attention score matrix in HBM. At sequence length $N = 4096$ with 32-bit floats, that matrix is already 64 MB — larger than the entire A100 SRAM. Every softmax operation, every masking step, every multiply with $V$ requires loading that matrix from HBM. The GPU's arithmetic units are fast enough to finish a tile computation in nanoseconds; the HBM fetch takes microseconds. The result is that the GPU spends the vast majority of wall-clock time //waiting// for data, not computing.

Memory bandwidth, not FLOPs, is therefore the true bottleneck for attention at long sequences. Standard attention is //memory-bound//: the number of HBM reads and writes is $\Theta(N^2)$ regardless of the arithmetic. Flash Attention is also memory-bound, but its HBM access pattern is $\Theta(N^2 d / M)$, where $d$ is head dimension — a factor of $M/d$ fewer round-trips. For typical values ($M \approx 20\text{ MB}$, $d = 64$) this is roughly a 5–10× reduction in HBM traffic.

---

!! What the Math Is Actually Doing

The obstacle to tiling attention is softmax. The standard formula $\text{softmax}(x)_i = e^{x_i} / \sum_j e^{x_j}$ requires the full row of logits $x$ before a single output can be produced, which is why naive tiling fails — you cannot normalise until you have seen everything.

The ''online softmax trick'' breaks this dependency. For a row $x = [x_1, \dots, x_N]$, we track two running scalars: the current maximum $m_i = \max(x_1, \dots, x_i)$ and the current sum $\ell_i = \sum_{j=1}^{i} e^{x_j - m_i}$. When a new tile $x_{i+1}, \dots, x_{i+B}$ arrives, we update:

$$m_{\text{new}} = \max(m_i,\; m_{\text{tile}})$$

$$\ell_{\text{new}} = e^{m_i - m_{\text{new}}} \cdot \ell_i + e^{m_{\text{tile}} - m_{\text{new}}} \cdot \ell_{\text{tile}}$$

The rescaling factor $e^{m_i - m_{\text{new}}}$ corrects the previously accumulated sum for the revised maximum. This relies on the identity:

$$\text{softmax}(x)_i = \frac{e^{x_i - m}}{\sum_j e^{x_j - m}}$$

where $m = \max(x)$. Subtracting the maximum before exponentiating prevents numerical overflow and cancels out of numerator and denominator — the value of $m$ is irrelevant to the final result, only the relative differences matter.

The output accumulator $O$ is similarly rescaled tile by tile:

$$O_{\text{new}} = \frac{\ell_i \cdot e^{m_i - m_{\text{new}}} \cdot O_{\text{old}} + e^{m_{\text{tile}} - m_{\text{new}}} \cdot O_{\text{tile}}}{\ell_{\text{new}}}$$

At the end of the pass, $O$ equals the exact output of standard attention. The algorithm uses $O(N d)$ HBM memory (linear in sequence length) rather than $O(N^2)$, enabling sequences that would otherwise exhaust GPU memory entirely.

---

!! Practical Impact

Flash Attention delivers ''2–4× wall-clock speedup'' over standard attention on typical transformer workloads — not because it does fewer FLOPs (it actually does slightly more due to recomputation during the backward pass) but because it is no longer starved of data. Benchmarks on GPT-style models show end-to-end training throughput improvements of 15–45% when replacing standard attention.

The ''sub-quadratic memory'' footprint is arguably more important than raw speed. Standard attention at $N = 32768$ requires ~4 GB for the attention matrix alone at 16-bit precision; Flash Attention holds only a few MB of tile buffers. This directly enabled practical training of models with 32K, 64K, and 128K context windows that were previously impossible to fit in a single GPU.

Flash Attention is now ''standard in major frameworks'': PyTorch 2.0 uses it by default via `torch.nn.functional.scaled_dot_product_attention`; JAX and TensorFlow have equivalent fused kernels; Hugging Face Transformers exposes it through the `attn_implementation="flash_attention_2"` flag.

''Flash Attention 2'' (Dao, 2023) improved parallelism by partitioning work across the sequence dimension in addition to the batch and head dimensions, achieving better GPU utilisation and roughly 2× the throughput of FA1 on long sequences. ''Flash Attention 3'' (Shah et al., 2024) targeted H100 hardware specifically, exploiting asynchronous memory copies (TMA instructions), warp-specialisation, and FP8 arithmetic to push utilisation past 75% of theoretical H100 peak — approximately 1.5–2× faster than FA2.

---

!! When Flash Attention Breaks Down

''Variable-length sequences with heavy padding:'' Flash Attention processes attention in tiles of fixed block size. When a batch contains sequences of wildly varying lengths padded to the same maximum, the tiling scheme performs redundant computation on padding tokens. The FA2 varlen API mitigates this by packing multiple sequences into a single flat buffer and using a cumulative sequence-length array to skip padding, but this complicates batching logic and is not universally supported in higher-level libraries.

''Custom attention patterns and sparse masks:'' Flash Attention's tiling is most efficient for dense or simple-structured masks (causal, sliding window). Arbitrary sparse attention masks — as used in some retrieval-augmented or graph-structured models — require per-tile mask evaluation that can break the regular memory access pattern, reducing or eliminating the IO advantage. Implementing a novel sparse pattern with Flash Attention typically requires writing a custom CUDA or Triton kernel, which is a significant engineering investment.

''Very short sequences or small batch sizes:'' The IO-bandwidth savings of Flash Attention are most pronounced when the $N \times N$ matrix would be large enough to cause significant HBM traffic. At short sequence lengths (below ~512 tokens) or with very small batch sizes, the overhead of the fused kernel and the reduced parallelism can make Flash Attention slower than a well-tuned unfused implementation. Standard benchmarks show that for $N \leq 128$, cuBLAS-backed naive attention is sometimes faster on certain hardware configurations.

!! Technical Details

''Key papers:'' Dao et al. (2022) "FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness" (NeurIPS 2022) introduced the tiling algorithm and online softmax. Dao (2023) "FlashAttention-2: Faster Attention with Better Parallelism and Work Partitioning" presented the sequence-parallel decomposition. Shah et al. (2024) "FlashAttention-3: Fast and Accurate Attention with Asynchrony and Low-precision" targeted H100 with TMA and FP8.

''SRAM block size tuning:'' The tile size $B_r \times B_c$ must satisfy $B_r \cdot B_c \leq M / 4$ (four arrays — Q tile, K tile, V tile, O tile — must fit simultaneously in SRAM). Optimal block sizes are hardware-specific; FA2 autotuning searches over common sizes (64, 128) at runtime.

''IO complexity:'' Standard attention performs $\Theta(N d + N^2)$ HBM reads/writes. Flash Attention performs $\Theta(N^2 d / M)$ reads/writes. For $d = 128$, $M = 20\text{ MB}$, $N = 4096$: standard ~268 MB of HBM traffic, Flash ~104 MB — roughly 2.6× less.

''Causal masking within tiles:'' The backward half of each tile (where column index exceeds row index) is masked to $-\infty$ before the softmax, which zeroes those exponentials. Tiles that lie entirely below the diagonal (all entries masked) are skipped, giving roughly 2× speedup for causal models on top of the IO savings.

''Grouped Query Attention (GQA) with Flash Attention:'' GQA reduces the number of KV heads relative to Q heads. FA2 implements this by having multiple Q tiles share the same K/V tile, reducing the number of K/V loads from HBM proportionally. For a 4:1 GQA ratio, KV HBM traffic is quartered with no loss of numerical precision.

''Hardware specifics:'' A100 SRAM = ~20 MB shared memory; H100 SRAM = ~50 MB; RTX 4090 SRAM = ~72 MB (consumer). H100 additionally has a second-level SRAM (L2 cache) of 50 MB. Larger SRAM allows larger tile sizes and further reduces HBM round-trips.

''Backward pass:'' Flash Attention avoids storing the $N \times N$ attention matrix for the backward pass by recomputing it from the saved Q, K, V tiles during backpropagation. This trades HBM memory for extra FLOPs — acceptable because the recomputation is fast and the memory saving is large.
