! Gradient Checkpointing
<img src="assets/gradient_checkpointing.svg" alt="Diagram comparing standard backpropagation (all activations stored) with gradient checkpointing (only checkpoint activations kept, rest recomputed)" style="max-width:100%; margin:1em 0;">

When hiking, you can carry a photo of every viewpoint so you can retrace your steps precisely — fast, but a heavy backpack. Or you can photograph only key landmarks and re-walk sub-paths from memory — lighter pack, some extra steps. Gradient checkpointing chooses the lighter pack: it saves only a subset of intermediate computations and recomputes the rest during backpropagation.

!! The Core Insight

Chen et al. (2016) observed that during backpropagation, the backward pass needs the activations produced during the forward pass. Standard training stores ''all'' activations — O(N) memory for N layers. Gradient checkpointing saves activations only at "checkpoint" layers (every √N layers) and recomputes intermediate activations on-the-fly during the backward pass.

The result: memory drops from O(N) to O(√N). The cost: roughly 33% additional compute (one extra forward pass through half the network on average). This makes training very large models feasible on limited hardware. Related: [[Backpropagation and Chain Rule]], [[Deep Learning]], [[Hardware Lottery]], [[Batch Normalization]], [[Gradient Descent Quiz]].

!! The Algorithm

''Forward pass:'' Run the full network normally, but only ''store'' activations at designated checkpoint layers (every k layers). Discard all other intermediate activations immediately after each layer.

''Backward pass:'' When the gradient computation needs the activation of a non-checkpoint layer, recompute it from the nearest checkpoint upstream by running a mini forward pass through just that segment. Then compute the gradient and discard again.

''Optimal k:'' Setting k = √N minimises peak memory (derived by minimising M(k) = k·B + N/k · C over k, where B is per-layer activation size and C is checkpoint size). In practice, you manually specify checkpoint layers or use PyTorch's built-in utility:

{{{torch.utils.checkpoint.checkpoint(function, *inputs)}}}

Wrapping any module or sequence of layers activates checkpointing for that segment.

!! What the Math Is Actually Doing

Let N = number of layers, B = memory per activation (batch × sequence length × hidden dim).

''Standard storage:'' $M_{\text{full}} = O(N \cdot B)$

''Checkpointing at interval k:'' At any moment during the backward pass you hold at most k consecutive activations (from the current recompute segment) plus the N/k checkpoint activations:

$$M_{\text{ck}}(k) = k \cdot B + \frac{N}{k} \cdot B$$

Minimise over k: $\frac{d}{dk}M_{\text{ck}} = B - \frac{N}{k^2} = 0 \Rightarrow k^* = \sqrt{N}$

Substituting: $M_{\text{ck}}(k^*) = 2\sqrt{N} \cdot B = O(\sqrt{N})$

''Compute overhead:'' Each non-checkpoint segment (length ~√N) is computed twice — once in the original forward pass and once during recompute. For k = √N, the total recomputed work ≈ one full forward pass through half the network. Since training normally does one forward + one backward ≈ two forward-equivalent passes, the overhead is ~0.5 / 2 ≈ 25–33% depending on backward-pass cost.

!! Practical Usage

''PyTorch API:''

{{{
import torch.utils.checkpoint as ckpt

output = ckpt.checkpoint(my_transformer_block, hidden_states)
}}}

''Typical strategy:'' checkpoint every Transformer block (attention + MLP together). Tradeoff: attention is expensive to recompute; MLP is cheap. Some practitioners checkpoint only attention to balance cost.

''Flash Attention synergy:'' Flash Attention already recomputes attention scores during its backward pass (to avoid storing the N×N attention matrix). Combining gradient checkpointing with Flash Attention gives additive memory savings.

''LoRA + gradient checkpointing:'' Together they make fine-tuning 70B-parameter models feasible on a single GPU — LoRA reduces trainable parameters, checkpointing cuts activation memory.

''Activation offloading:'' A complementary technique — move checkpoint activations to CPU RAM instead of discarding them, at the cost of PCIe transfer latency. Slower than recompute for cheap layers, faster for expensive ones.

!! When Gradient Checkpointing Breaks Down

''Non-deterministic layers:'' If a layer uses Dropout and the RNG state differs between the original forward pass and the recompute, gradients are incorrect. Fix: PyTorch's checkpoint utility automatically saves and restores the RNG state, but custom implementations must do this explicitly.

''Custom CUDA kernels:'' The recomputed result must be bit-identical to the original. Kernels that use non-associative floating-point reductions or depend on launch-time state can produce different values, silently corrupting gradients.

''Excessive recompute overhead:'' If checkpoint intervals are too small (k ≪ √N), compute overhead grows without bound. For very deep networks or when each layer is extremely expensive (e.g., mixture-of-experts routing), the 33% overhead assumption breaks down and wall-clock time can double.

!! Technical Details

''Foundational papers:''
- Chen et al. 2016 — "Training Deep Nets with Sublinear Memory Cost" — introduced the √N checkpoint scheme.
- Gruslys et al. 2016 — "Memory-Efficient Backpropagation Through Time" — extended to RNNs with dynamic programming for optimal checkpoint placement.

''PyTorch implementation notes:'' `torch.utils.checkpoint` works with `torch.autograd` by registering a custom backward function that triggers recompute. Supports `use_reentrant=False` (recommended in PyTorch ≥ 2.0) for better compatibility with `torch.compile`.

''Mixed precision:'' Recomputed activations use the same dtype as the original forward pass. If training in bf16, recomputed values are also bf16 — no precision mismatch.

''Distributed training:'' DeepSpeed ZeRO combines gradient checkpointing with parameter sharding across GPUs. ZeRO-3 shards parameters, gradients, and optimizer states; gradient checkpointing reduces per-GPU activation memory independently.

''Selective checkpointing:'' Research (e.g., Selective Activation Recomputation, Korthikanti et al. 2022) identifies which layers are cheapest to recompute and checkpoints only the expensive ones, reducing overhead below the 33% baseline.
