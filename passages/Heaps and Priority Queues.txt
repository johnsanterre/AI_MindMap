! Heaps and Priority Queues

<img src="assets/heaps_and_priority_queues.svg" alt="Diagram showing a binary min-heap as a tree with nodes 1,3,5,7,9,8,12, its array representation, and key operations with their time complexities" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine an emergency room where patients are not seen in the order they arrive, but in order of severity — the most critical case always goes first, even if they walked in last. That is exactly what a ''priority queue'' does with data. And the most elegant way to build one is a ''heap'': a tree-shaped data structure that keeps its most important element always sitting at the very top, ready to be grabbed in constant time. Heaps are deceptively simple — just an array with a clever indexing trick — yet they power some of the most important algorithms in computer science and machine learning, from [[Shortest Path Algorithms]] to beam search in language models.

!! The Core Insight

A ''binary heap'' is a complete binary tree stored as a flat array, where every parent is smaller (in a min-heap) or larger (in a max-heap) than its children. This single rule — the ''heap property'' — guarantees that the minimum (or maximum) element is always at the root, retrievable in $O(1)$ time. When you insert or remove an element, you restore the heap property by "bubbling" elements up or down the tree, which takes at most $O(\log n)$ swaps because the tree has $\log n$ levels.

The genius is the array trick: node at index $i$ has children at $2i+1$ and $2i+2$, and its parent at $\lfloor(i-1)/2\rfloor$. No pointers, no wasted memory — just arithmetic on array indices. This makes heaps extraordinarily cache-friendly compared to pointer-based trees like [[Binary Search Trees]].

A ''priority queue'' is the abstract interface — "give me the highest-priority item" — and a heap is the standard concrete implementation. The distinction matters: you could implement a priority queue with a sorted list (slow insert) or an unsorted list (slow extraction), but heaps give $O(\log n)$ for both, which is the optimal tradeoff for comparison-based structures.

!! Heap Variants and Extensions

The basic binary heap is just the beginning. Several variants exist for specialized use cases:

''d-ary heaps'' generalize binary heaps to $d$ children per node. The tree becomes shallower (height $\log_d n$), speeding up bubble-up to $O(\log_d n)$ but slowing bubble-down to $O(d \log_d n)$ since you must compare against all $d$ children. A 4-ary heap often outperforms a binary heap in practice for Dijkstra's algorithm because decrease-key (bubble-up) is more frequent than extract-min.

''Fibonacci heaps'' achieve $O(1)$ amortized time for insert and decrease-key, and $O(\log n)$ amortized for extract-min. This makes them theoretically optimal for algorithms like Dijkstra and Prim's [[Minimum Spanning Trees]] algorithm, reducing their runtime from $O((V+E)\log V)$ to $O(E + V\log V)$. In practice, the constant factors and complexity of Fibonacci heaps often make simpler binary heaps faster for moderate graph sizes.

''Binomial heaps'' support efficient merging of two heaps in $O(\log n)$ time — something binary heaps cannot do without $O(n)$ work. ''Pairing heaps'' offer a simpler alternative to Fibonacci heaps with excellent empirical performance, though their theoretical bounds are not as tight.

''Min-max heaps'' support both find-min and find-max in $O(1)$ by alternating min-levels and max-levels in the tree. This is useful when you need access to both extremes, such as maintaining a sliding window of values.

!! What the Math Is Actually Doing

The heap property for a min-heap can be stated formally:

$$\forall i > 0: A\left[\left\lfloor\frac{i-1}{2}\right\rfloor\right] \leq A[i]$$

Every element is at most as large as its descendants. This is a ''partial order'' — unlike a [[Binary Search Trees|binary search tree]], a heap does not maintain a total order among siblings. The left child can be larger or smaller than the right child. This relaxation is what makes heaps faster to maintain: you only need to fix one path from root to leaf, not keep the entire structure sorted.

''Build-heap'' (Floyd's algorithm) runs in $O(n)$ despite each sift-down taking up to $O(\log n)$. The key insight is that most nodes are near the bottom and have very little to sift. Formally:

$$T(n) = \sum_{h=0}^{\lfloor \log n \rfloor} \left\lceil \frac{n}{2^{h+1}} \right\rceil \cdot O(h) = O\!\left(n \sum_{h=0}^{\infty} \frac{h}{2^h}\right) = O(n)$$

The sum $\sum_{h=0}^{\infty} h/2^h = 2$ converges, giving linear time. This is why heapsort's first phase (building the heap) costs only $O(n)$, and the $O(n \log n)$ cost comes entirely from the $n$ extract-min operations.

''Decrease-key'' — lowering the priority of an element already in the heap — requires knowing the element's position. In a binary heap, this takes $O(\log n)$ via bubble-up. In a Fibonacci heap, it takes $O(1)$ amortized by cutting the node from its parent and adding it to the root list, using a clever ''cascading cut'' mechanism.

!! Heaps in Algorithms and ML

Heaps are the backbone of several algorithms central to machine learning:

''Dijkstra's algorithm'' for [[Shortest Path Algorithms]] maintains a priority queue of unvisited nodes, always expanding the closest one. With a binary heap, this runs in $O((V + E) \log V)$. Every graph-based ML method that needs shortest paths — from knowledge graph embeddings to network analysis — relies on this.

''Beam search'' in sequence-to-sequence models and large language models uses a priority queue to maintain the top-$k$ partial hypotheses at each step. Only the $k$ highest-scoring beams survive to the next timestep, and a heap efficiently identifies which beams to keep.

''Heapsort'' is an in-place, $O(n \log n)$ [[Sorting Algorithms|sorting algorithm]] with guaranteed worst-case performance — unlike quicksort, which degrades to $O(n^2)$ on adversarial inputs. Build a max-heap, then repeatedly extract the maximum. It is not stable and has poor cache behavior compared to mergesort, but its $O(1)$ extra space is sometimes decisive.

''Top-K selection'' — finding the $k$ largest elements from $n$ candidates — takes $O(n \log k)$ with a min-heap of size $k$. Scan the data once, maintaining a heap of the $k$ largest seen so far. Each new element is compared to the heap's minimum; if larger, it replaces the min via extract-and-insert. This is used in nearest-neighbor search, recommendation systems, and selecting the top-$k$ predictions from a model.

''Huffman coding'' for data compression greedily merges the two lowest-frequency symbols using a min-heap, building an optimal prefix-free code. The algorithm runs in $O(n \log n)$ for $n$ symbols.

''Prim's algorithm'' for [[Minimum Spanning Trees]] uses a priority queue identically to Dijkstra, but keyed on edge weights rather than distances.

!! When Heaps and Priority Queues Break Down

''1. Search is expensive.'' A heap can find the minimum in $O(1)$ and insert/extract in $O(\log n)$, but //searching// for an arbitrary element takes $O(n)$ — you must scan the entire array. If you frequently need to look up whether a specific element is present or find its current priority, a heap alone is insufficient. You need an auxiliary hash map to track positions, which adds complexity. When search is the dominant operation, a [[Binary Search Trees|balanced BST]] is more appropriate, offering $O(\log n)$ for search, insert, and delete.

''2. Merging two heaps is slow for binary heaps.'' Combining two binary heaps of sizes $m$ and $n$ requires $O(m + n)$ — essentially rebuilding from scratch. If your application frequently merges priority queues (e.g., parallel graph algorithms, external memory sorting), binary heaps are a poor fit. Binomial heaps merge in $O(\log n)$ and Fibonacci heaps in $O(1)$.

''3. Decrease-key with position tracking.'' Many graph algorithms (Dijkstra, Prim) need to //update// the priority of an element already in the heap. A basic binary heap provides no way to find an element by key — you need a separate index mapping elements to their heap positions, and you must maintain this index through every swap. This bookkeeping is error-prone and adds constant-factor overhead. Fibonacci heaps solve this elegantly in theory, but their practical complexity often makes indexed binary heaps the pragmatic choice.

!! Technical Details

''Time complexities (binary min-heap):''
- Insert: $O(\log n)$ worst-case
- Extract-min: $O(\log n)$ worst-case
- Peek (find-min): $O(1)$
- Build-heap: $O(n)$ via Floyd's algorithm
- Decrease-key: $O(\log n)$ with index tracking
- Delete arbitrary: $O(\log n)$ with index tracking
- Search: $O(n)$

''Fibonacci heap (amortized):''
- Insert: $O(1)$
- Extract-min: $O(\log n)$
- Decrease-key: $O(1)$
- Merge: $O(1)$

''Space:'' $O(n)$ for all heap variants. Binary heaps use a contiguous array with zero pointer overhead — just $n$ elements.

''Heapsort'' is not stable (equal elements may be reordered). Its $O(n \log n)$ worst-case is better than quicksort's worst case, but quicksort is faster on average due to better cache locality and smaller constant factors.

''Heap as an implicit data structure:'' because a complete binary tree of $n$ nodes has a unique shape, no structural metadata is stored — the tree topology is implicit in the array indices. This is why heaps are one of the most space-efficient tree structures.

''Python:'' `heapq` module provides a min-heap. For max-heap, negate values. `heapq.heappush()`, `heapq.heappop()`, `heapq.heapify()` (in-place, $O(n)$), `heapq.nlargest(k, iterable)`.

Related topics: [[Sorting Algorithms]], [[Binary Search Trees]], [[Shortest Path Algorithms]], [[Minimum Spanning Trees]].
