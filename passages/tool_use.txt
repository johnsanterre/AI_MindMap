! Tool Use

<img src="assets/tool_use.svg" alt="Diagram showing how an LLM generates structured function calls to external tools and receives results back into context" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you are brilliant at writing essays but cannot do long division. You could struggle through the arithmetic yourself — getting it wrong half the time — or you could pick up a calculator. ''Tool use'' in large language models is exactly that move. Instead of forcing the model to hallucinate a computation, retrieve a live fact from memory, or execute code in its head, the system lets the model //call an external tool// — a calculator, a search engine, a Python interpreter, a database, an API — receive the result, and weave it into its response. The model does not become smarter at arithmetic; it becomes smart enough to know when to delegate. This single architectural decision transforms a text generator into something far more capable: a reasoning system that knows the boundaries of its own competence and can reach beyond them.

---

!! The Core Insight

A language model is a conditional text generator. It is very good at predicting plausible next tokens, but there are entire classes of tasks where plausible and correct diverge sharply: exact arithmetic, real-time information retrieval, code execution, structured database queries. ''Tool use'' bridges this gap by giving the model a vocabulary of //actions// in addition to its vocabulary of //words//. When the model encounters a sub-problem better handled by an external system, it emits a structured ''function call'' — a JSON object specifying which tool to invoke and what arguments to pass — instead of generating a free-text answer. The runtime intercepts this call, executes the tool, and injects the result back into the model's context as an ''observation''. The model then continues generating, now grounded in a verified fact or computation rather than a guess.

The key insight is that function calling does not require a fundamentally new architecture. It emerges from the same next-token prediction objective that produces fluent text. Models trained on code naturally acquire the ability to emit structured JSON, and models fine-tuned on examples of (query, tool call, observation, response) trajectories learn to reliably decide //when// and //how// to invoke tools. This is closely related to <<link "Instruction Tuning" "Instruction Tuning">><</link>>, which teaches models to follow structured directives, and to <<link "In-Context Learning" "In-Context Learning">><</link>>, which allows models to pick up new tool schemas from examples in the prompt. Tool use is the foundation on which <<link "LLM Agents" "LLM Agents">><</link>> are built — an agent is essentially a model using tools inside a reasoning loop.

---

!! The Function Calling Protocol

The modern tool-use interface works through a well-defined protocol. Before generation begins, the system prompt includes a set of ''tool definitions'' — each specifying a function name, a natural-language description, and a parameter schema (typically JSON Schema). The model sees these definitions as part of its context, just like any other text.

During generation, the model may decide to emit a ''tool call'' rather than ordinary text. In practice this means the model produces a structured JSON block:

{{{
{"name": "search", "arguments": {"query": "population of France 2024"}}
}}}

The runtime detects this structured output, routes it to the appropriate tool implementation, executes it, and returns the result as a new message of type ''tool result'' or ''observation''. The model reads this result and continues generating. It may call additional tools, chain results together, or produce a final text response that synthesises everything.

This protocol is ''stateless from the tool's perspective'' — tools do not remember prior calls unless explicitly designed to — but ''stateful from the model's perspective'', because every tool call and result accumulates in the context window. The model can refer back to prior observations when making decisions about subsequent calls. This is what enables multi-step reasoning: search for a fact, use it as input to a calculation, verify the calculation by running code, then compose the final answer. The approach is central to <<link "Prompt Engineering" "Prompt Engineering">><</link>>, where careful tool descriptions dramatically affect reliability.

---

!! What the Math Is Actually Doing

At each generation step, the model produces a probability distribution over its vocabulary, which now includes both regular tokens and special ''tool call tokens'' (or, equivalently, the model is trained to produce JSON-formatted tool calls as token sequences). Let $V$ be the token vocabulary and $T$ be the set of valid tool call formats. The model computes:

$$p(y_t \mid y_{<t}, x, D) = \text{softmax}(W h_t)$$

where $x$ is the user query, $D = \{d_1, \ldots, d_k\}$ is the set of tool definitions in the prompt, and $h_t$ is the hidden state at position $t$. The model implicitly learns a ''routing function'' $\sigma: \mathcal{X} \times \mathcal{D} \to \{0, 1\}$ that decides whether the current context warrants a tool call ($\sigma = 1$) or a direct text response ($\sigma = 0$).

Training the routing function uses supervised fine-tuning on curated datasets of (prompt, tool_call, observation, response) tuples. The loss is standard cross-entropy over the entire sequence including the tool call tokens:

$$\mathcal{L} = -\sum_{t=1}^{T} \log p_\theta(y_t \mid y_{<t})$$

where the target sequence $y$ includes both the tool call JSON and the final response. Some systems additionally use <<link "Reinforcement Learning" "Reinforcement Learning">><</link>> — specifically RLHF or reward-model scoring — to improve the model's ability to decide //when// tool use is necessary versus when direct generation suffices. The reward signal penalises unnecessary tool calls (latency cost) and rewards correct tool use that produces verifiably accurate answers.

For multi-turn tool use, the context grows as $s_{t+1} = s_t \oplus a_t \oplus o_t$, where $a_t$ is the tool call at step $t$ and $o_t$ is the returned observation. The model conditions on this full history at each step, enabling it to chain tool calls: $a_2$ can depend on $o_1$, and so on. This sequential dependency structure is what separates tool use from simple template filling — the model dynamically decides the //next// tool call based on //all prior// results.

---

!! Tool Design and Practical Patterns

The quality of tool definitions matters as much as model capability. Empirical studies show that improving tool descriptions — adding precise parameter types, boundary conditions, example calls, and explicit error formats — can increase success rates by 20-30 percentage points with no change to the model. A good tool definition has five properties: (1) an unambiguous name, (2) a one-sentence description of what it does, (3) typed parameters with constraints, (4) example invocations, and (5) structured error returns.

''Parallel tool calling'' allows the model to emit multiple tool calls simultaneously when they are independent. If the user asks "What is the weather in Paris and the population of Tokyo?", the model can call both tools in one generation step rather than sequentially — halving latency. This requires the runtime to support batched execution and the model to be trained on parallel-call examples.

''Tool result verification'' is an emerging pattern where the model is trained to be sceptical of tool returns. Rather than blindly incorporating an observation, the model checks whether the result is plausible given the query. If a calculator returns a negative number for a factorial, or a search returns a result from an unreliable source, the model can flag the discrepancy or retry with a refined query. This pattern is closely related to the self-correction loops studied in <<link "LLM Agents" "LLM Agents">><</link>>.

''Fallback generation'' handles cases where a tool call fails (network timeout, rate limit, malformed response). Well-designed systems allow the model to detect the failure, explain it to the user, and either retry or fall back to its parametric knowledge with an appropriate disclaimer.

---

!! When Tool Use Breaks Down

''Hallucinated tool calls:'' The model generates a call to a tool that does not exist, uses fabricated parameter names, or passes arguments that violate the schema. This happens most often when the tool definitions are vague, when the model encounters a task outside its training distribution, or when the context is very long and the tool definitions have scrolled far from the current generation point. The model confidently emits syntactically valid JSON for a non-existent function, and the runtime returns an error that the model may then misinterpret or ignore. Mitigation requires strict schema validation, clear error messages injected back into context, and constrained decoding that restricts output to valid tool names.

''Over-reliance on tools (tool addiction):'' Some fine-tuned models learn to call tools even when the answer is trivially available in their parametric knowledge. Asked "What is 2 + 2?", the model routes to a calculator rather than answering directly. This adds unnecessary latency and API cost. The root cause is training data that over-represents tool-call trajectories relative to direct-answer trajectories. The fix is balanced training data that includes examples where the correct behaviour is to answer directly, and reward signals that penalise unnecessary tool invocations.

''Misinterpretation of tool results:'' The tool returns a correct result, but the model misreads or misinterprets it. A search engine returns a snippet saying "The population declined to 65 million" and the model extracts "65 million" while ignoring the word "declined," producing a misleadingly optimistic answer. This failure mode is especially dangerous because the tool result is objectively correct — the error is in the model's post-processing. It is closely related to reading comprehension failures and is exacerbated by long contexts where the observation is far from the final generation point.

!! Technical Details

''Foundational work:'' Schick et al. (2023), "Toolformer: Language Models Can Teach Themselves to Use Tools." Trained a model to self-annotate its own training data with tool calls by identifying positions where a tool would improve prediction loss, then fine-tuning on the augmented data. This was the first demonstration that tool use could be learned in a self-supervised manner rather than requiring hand-curated training data.

''API milestones:'' OpenAI introduced function calling for GPT-4 in June 2023, providing a structured JSON interface for tool definitions and calls. Anthropic followed with tool use in Claude (2023-2024). Google's Gemini added function calling in late 2023. All three follow a similar protocol: tool definitions in the system prompt, structured JSON tool calls in the assistant turn, tool results in a dedicated message role.

''Gorilla (Patil et al. 2023):'' Fine-tuned LLaMA on a dataset of 1,645 API calls across Torch Hub, TensorFlow Hub, and HuggingFace. Demonstrated that specialised fine-tuning dramatically improves API call accuracy, with Gorilla outperforming GPT-4 on correct API selection while hallucinating fewer non-existent endpoints.

''Training approaches:'' Three main strategies exist. (1) ''Supervised fine-tuning'' on curated (prompt, tool_call, result, response) tuples — most common and reliable. (2) ''Self-supervised annotation'' (Toolformer style) — the model identifies where tools would help and generates its own training data. (3) ''RL-based optimisation'' — using reward models that score trajectory-level outcomes (did the final answer improve because of the tool call?) to optimise the tool-use policy via <<link "Reinforcement Learning" "Reinforcement Learning">><</link>>.

''Constrained decoding:'' Production systems often use grammar-constrained generation to ensure tool calls are valid JSON matching the declared schema. This eliminates syntactic hallucinations (malformed JSON, wrong parameter types) and restricts tool names to the declared set, at the cost of slightly increased inference complexity.

''Benchmarks:'' ''API-Bank'' (Li et al. 2023) evaluates tool-use accuracy across 73 API tools. ''ToolBench'' (Qin et al. 2023) provides 16,000+ real-world APIs from RapidAPI for evaluating multi-step tool use. ''BFCL'' (Berkeley Function Calling Leaderboard) tracks structured function calling accuracy across models.
