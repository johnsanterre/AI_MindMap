! Complexity Classes

<img src="assets/complexity_classes.svg" alt="Nested diagram showing complexity class hierarchy: P inside NP inside PSPACE inside EXPTIME, with NP-complete at the NP boundary" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Some puzzles are easy to solve; others are easy to //check// but fiendishly hard to find. Consider a Sudoku puzzle: verifying a completed grid takes seconds — just scan rows, columns, and boxes. But finding the solution from a blank grid can take hours of exhaustive search. This gap between //checking// and //finding// is the heart of complexity theory. ''Complexity classes'' categorize problems by how difficult they fundamentally are — not just for today's computers, but for //any// computer imaginable, running for any amount of time, using any algorithm. The question of whether these categories truly differ is the deepest unsolved problem in all of computer science.

!! The Core Insight

The central drama of complexity theory is the ''P vs NP'' question. ''P'' (polynomial time) contains problems a computer can //solve// quickly — in time bounded by a polynomial function of the input size, like $O(n^2)$ or $O(n^3)$. Sorting a list, finding shortest paths, matrix multiplication — all are in P.

''NP'' (nondeterministic polynomial time) contains problems whose solutions can be //verified// quickly, even if finding them seems hard. Given a Sudoku solution, you can check it in $O(n^2)$ time. Given a proposed route for a traveling salesman, you can verify its length immediately. NP includes P — anything you can solve quickly, you can also verify quickly — but it may contain much harder problems too.

The $P = NP$ question asks: if you can check a solution quickly, can you always //find// one just as quickly? If yes, then cryptography collapses, drug discovery becomes trivial, and optimization becomes easy. Almost every researcher believes $P \neq NP$ — that finding is genuinely harder than checking — but ''no one has proven it''. This is the greatest open problem in computer science, and one of the Millennium Prize Problems (worth $1,000,000).

!! The Class Hierarchy

Complexity classes form a nested hierarchy, each containing the one before:

$$P \subseteq NP \subseteq PSPACE \subseteq EXPTIME$$

''P'' — Problems solvable in polynomial time. Examples: sorting, shortest path (Dijkstra), linear programming, primality testing.

''NP'' — Problems with polynomial-time verifiable solutions. Examples: Sudoku, Boolean satisfiability (SAT), graph coloring, subset sum, the traveling salesman decision problem.

''NP-complete'' — The //hardest// problems in NP. Every other NP problem can be reduced to them. If you could solve any NP-complete problem in polynomial time, you could solve //all// of NP in polynomial time — so $P = NP$. Examples: SAT, 3-coloring, Hamiltonian cycle, knapsack.

''NP-hard'' — At least as hard as any NP-complete problem, but not necessarily //in// NP (solutions may not even be verifiable in polynomial time). Examples: the halting problem, optimal neural network training.

''Co-NP'' — The complements of NP problems. Instead of "does a solution exist?", ask "does //no// solution exist?" Tautology checking is Co-NP-complete. Whether NP = Co-NP is another open question.

''#P'' (sharp-P) — Counting the //number// of solutions to an NP problem, not just detecting one. Counting satisfying assignments of a Boolean formula is #P-complete — often harder than the corresponding NP decision problem.

''PSPACE'' — Problems solvable with polynomial //space// (memory), regardless of time. Includes two-player game strategies (e.g., generalized chess positions, QBF). PSPACE contains both NP and Co-NP.

''EXPTIME'' — Problems solvable in exponential time $O(2^{n^k})$. Chess on an $N \times N$ board is EXPTIME-complete.

''BPP'' (bounded-error probabilistic polynomial time) — Problems solvable by a randomized algorithm in polynomial time with error probability at most 1/3. Many believe BPP = P, but this is unproven. Randomized algorithms for primality testing (before AKS) lived here.

!! What the Math Is Actually Doing

The formal backbone is the ''Turing machine'' — an abstract model of computation with an infinite tape and a finite set of states. Polynomial time means the machine halts in $p(n)$ steps for some polynomial $p$, where $n$ is the input length.

''Polynomial-time many-one reductions'' are the key tool. We write $A \leq_p B$ (read: "A reduces to B in polynomial time") to mean: given any instance of problem A, we can transform it into an instance of B in polynomial time, such that A has a solution iff B does. If $A \leq_p B$ and B is easy, then A is easy. Contrapositively, if A is hard and $A \leq_p B$, then B is at least as hard as A.

The ''Cook-Levin theorem'' (1971) established that Boolean satisfiability (SAT) is NP-complete: every NP problem reduces to SAT in polynomial time. The proof shows that the computation of any nondeterministic polynomial-time Turing machine can be encoded as a Boolean formula.

$$\text{SAT is NP-complete} \iff \forall L \in NP,\ L \leq_p \text{SAT}$$

Richard Karp (1972) then showed 21 other natural combinatorial problems are also NP-complete via chains of reductions: SAT $\leq_p$ 3-SAT $\leq_p$ graph 3-coloring $\leq_p$ clique $\leq_p$ vertex cover $\leq_p$ \ldots Each reduction is a polynomial-time transformation showing the new problem is at least as hard as the previous.

NP-hardness of neural network training: Blum and Rivest (1992) proved that training even a simple 3-node neural network to correctly classify a set of points is NP-hard. The reduction is from the problem of finding a consistent linear threshold function on a subset of bits.

!! Complexity in ML

Complexity theory has direct, practical consequences for machine learning:

''NP-hard training.'' In the worst case, training a neural network — finding weights that minimize empirical loss to zero — is NP-hard (Blum & Rivest 1992). Yet in practice, gradient descent on overparameterized networks works remarkably well. The gap between worst-case and typical-case complexity is enormous.

''Feature selection.'' Choosing the optimal subset of $k$ features from $d$ features to maximize predictive accuracy is NP-hard in general (equivalent to subset sum variants). In practice, greedy forward/backward selection or LASSO-style regularization gives tractable approximations.

''Combinatorial ML problems.'' Boolean satisfiability underlies constraint satisfaction in probabilistic graphical models, MAP inference in Markov Random Fields, and structured prediction. Many clustering problems (e.g., optimal k-means with $k > 1$ cluster) are NP-hard.

''Approximation algorithms.'' When exact solutions are NP-hard, the right question is: how close to optimal can we get in polynomial time? Approximation algorithms guarantee solutions within a factor $\alpha$ of optimal. For example, a 2-approximation for vertex cover always finds a solution at most twice the optimal size.

''Hardness of approximation.'' The ''PCP theorem'' (probabilistically checkable proofs, 1992–1998) showed that some NP-hard problems can't be approximated beyond a constant factor in polynomial time (assuming $P \neq NP$). MAX-3SAT cannot be approximated beyond a ratio of 7/8. This rules out even near-optimal solutions efficiently.

''Hyperparameter search.'' Neural architecture search (NAS) and hyperparameter optimization over discrete spaces are combinatorial problems. Their NP-hard structure motivates heuristics (random search, Bayesian optimization, evolutionary strategies) rather than exact solvers. !! When Complexity Classes Break Down

''1. Worst-case vs. average-case complexity.'' NP-hardness is a worst-case statement — there //exist// hard instances, but //typical// instances may be easy. Most real neural network training problems that are NP-hard in theory are tractable in practice. Random 3-SAT instances near the phase transition (clause-to-variable ratio ~4.27) are hard; most instances far from the transition are trivial. Complexity class membership says nothing about the instances you'll actually encounter.

''2. Constant factors hidden by Big O.'' Polynomial time doesn't mean //fast//. An algorithm running in $O(n^{10})$ is technically in P, but for $n = 100$ this is $10^{20}$ operations — utterly impractical. Conversely, an $O(2^n)$ algorithm with a tiny constant might run faster than an $O(n^3)$ algorithm for all problem sizes that arise in practice. The complexity class is an asymptotic statement; practical performance depends on the constants and the actual input sizes you face.

''3. Quantum complexity.'' Quantum computers operate in ''BQP'' (bounded-error quantum polynomial time). Shor's algorithm factors integers in quantum polynomial time — a problem believed outside classical BPP. But BQP almost certainly doesn't contain all of NP: there is strong evidence that quantum computers cannot solve NP-complete problems in polynomial time. Quantum speedups for ML (quantum linear algebra, Grover search) are real but more modest than sometimes claimed, and current quantum hardware is far from the scale needed.

!! Technical Details

''Key results:''
- Cook (1971): SAT is NP-complete (Cook-Levin theorem)
- Karp (1972): 21 combinatorial problems are NP-complete
- Ladner (1975): If $P \neq NP$, there exist problems in NP that are neither in P nor NP-complete (NP-intermediate). Integer factorization is a candidate.
- PCP theorem (Arora, Lund, Motwani, Sudan, Szegedy, 1992–1998): NP = PCP[O(log n), O(1)], implying hardness of approximation
- Blum & Rivest (1992): Training a 3-node NN is NP-hard

''Class relationships (known):''
$$P \subseteq NP,\quad NP \subseteq PSPACE,\quad PSPACE \subseteq EXPTIME$$
$$P \subsetneq EXPTIME \text{ (strict, by time hierarchy theorem)}$$

Whether $P \subsetneq NP$, $NP \subsetneq PSPACE$, or $PSPACE \subsetneq EXPTIME$ are all open.

''PSPACE'' includes: quantified Boolean formula (QBF), two-player game strategy, planning in partially observable MDPs.

''EXPTIME'' includes: chess/Go on $N \times N$ boards, generalized versions of many board games.

''Oracle separations:'' Baker, Gill, and Solovay (1975) showed there exist oracles relative to which $P = NP$ and oracles relative to which $P \neq NP$. This means any proof of $P \neq NP$ cannot relativize — it must use non-relativizing techniques, a major constraint on proof strategies.
