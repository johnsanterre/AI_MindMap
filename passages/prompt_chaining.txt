! Prompt Chaining

<img src="assets/prompt_chaining.svg" alt="Diagram illustrating prompt chaining: decomposing a complex task into sequential LLM steps where each output feeds the next" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Think of how a factory assembly line works. No single worker builds an entire car — one team welds the frame, passes it to a team that installs the engine, who passes it to a team that wires the electronics. Each station does one thing well, inspects its own work, and hands off a clean intermediate product. ''Prompt chaining'' applies the same principle to language model tasks. Instead of asking one massive prompt to do everything — research, reason, format, verify — you break the task into a sequence of focused prompts. Each prompt receives the output of the previous step as its input, performs one well-defined transformation, and produces a clean intermediate result. The final output emerges from the end of the chain, and if something goes wrong, you can inspect exactly which link failed.

---

!! The Core Insight

A single complex prompt asks the model to juggle multiple cognitive demands simultaneously: understanding context, reasoning about it, formatting output, checking constraints, and producing polished prose. This is like asking someone to write a book review while simultaneously translating it into French and ensuring it fits exactly 500 words — the competing demands interfere with each other, and the quality of each subtask degrades. Prompt chaining eliminates this interference by serializing the demands.

The key insight is that ''decomposition improves reliability more than it improves capability''. A sufficiently large model //can// sometimes handle the monolithic prompt. But it does so unreliably — sometimes it nails every requirement, sometimes it silently drops one. Chaining makes success reproducible. Each step's prompt can be short, specific, and tested independently. You can validate intermediate outputs before they propagate downstream. And when the chain fails, the error is localized to a specific step rather than buried in an opaque 2,000-token output.

This connects directly to <<link "Chain of Thought" "Chain of Thought">><</link>> prompting, which externalizes reasoning steps //within// a single prompt. Prompt chaining goes further: each reasoning step gets its own prompt, its own system instruction, and potentially its own model configuration. See also <<link "Prompt Engineering" "Prompt Engineering">><</link>> for the single-prompt techniques that chaining builds upon, and <<link "LLM Agents" "LLM Agents">><</link>> for the more autonomous extension where the model itself decides what step comes next.

---

!! Designing Effective Chains

The art of prompt chaining lies in choosing where to cut. A good decomposition has three properties: each step is ''independently testable'' (you can verify its output without running the full chain), each step's output is a ''clean interface'' (structured enough that the next step can parse it reliably), and the number of steps is ''minimal'' (every additional link adds latency and potential failure).

''Linear chains'' are the simplest pattern: Step 1 produces output $A$, Step 2 receives $A$ and produces $B$, Step 3 receives $B$ and produces $C$. Example: extract key claims from a document, then evaluate each claim for accuracy, then write a summary incorporating the evaluations. Each step receives only what it needs.

''Branching chains'' split a task into parallel subtasks that are later merged. Example: given a research question, one branch searches academic papers while another searches news articles. A final synthesis step receives both sets of results and produces a unified answer. Branching reduces total latency because parallel steps run simultaneously.

''Conditional chains'' include routing logic: a classifier step examines the input and decides which downstream chain to invoke. Example: a customer service system first classifies the message as billing, technical, or general, then routes to a specialized prompt chain for that category. This avoids one-size-fits-all prompts that handle every case poorly.

''Iterative chains'' loop: a generation step produces output, a verification step checks it against criteria, and if it fails, the generation step reruns with the verification feedback appended. This is closely related to the ''Reflexion'' pattern described in <<link "LLM Agents" "LLM Agents">><</link>> and is one of the most powerful patterns for achieving high-quality outputs.

---

!! What the Math Is Actually Doing

Consider a complex task $T$ that can be decomposed into $n$ sequential subtasks $T_1, T_2, \ldots, T_n$. Each subtask $T_i$ is handled by a prompt $P_i$ that takes the output of the previous step as input:

$$y_i = \text{LLM}(P_i, y_{i-1}), \quad y_0 = x_{\text{input}}$$

The final output is $y_n$. If each step succeeds with probability $p_i$, the overall chain success probability is:

$$P(\text{chain succeeds}) = \prod_{i=1}^{n} p_i$$

This looks alarming — more steps means lower reliability. But the key comparison is against the monolithic prompt. If the monolithic prompt succeeds with probability $p_{\text{mono}}$, chaining is worthwhile when:

$$\prod_{i=1}^{n} p_i > p_{\text{mono}}$$

In practice, this inequality holds because focused prompts have much higher per-step reliability than the monolithic alternative. A 3-step chain where each step succeeds 95% of the time has overall reliability $0.95^3 \approx 0.86$. If the monolithic prompt only succeeds 60% of the time, chaining is substantially better despite introducing more failure points.

The information-theoretic perspective: each intermediate output $y_i$ acts as a ''bottleneck'' that compresses the relevant information from $y_{i-1}$ for the task at hand. This is analogous to the ''information bottleneck'' principle — by forcing the chain to produce a clean intermediate representation, you discard noise and irrelevant detail that would otherwise accumulate in the model's context and confuse later reasoning.

When validation is added between steps, the effective per-step reliability increases. If a validator catches and reruns failed steps with probability $q$, the effective success rate per step becomes:

$$p_i^{\text{eff}} = p_i + (1 - p_i) \cdot q \cdot p_i = p_i(1 + q - q \cdot p_i)$$

For $p_i = 0.9$ and $q = 0.8$, this gives $p_i^{\text{eff}} \approx 0.98$, dramatically improving chain reliability.

!! Prompt Chaining in Production Systems

Real-world prompt chaining systems share several engineering patterns. ''Structured intermediate formats'' — JSON, XML, or typed schemas — make outputs machine-parseable and reduce the fragility of passing free-text between steps. A step that produces a JSON array of claims is far more reliable as input to the next step than a step that produces a paragraph mentioning the claims in prose.

''Context management'' is critical. Naive chaining passes the entire output of step $i$ as input to step $i+1$. For long chains, this means the context grows monotonically and eventually exceeds the model's window. Production systems selectively pass only the relevant portions of each intermediate output, sometimes using a summarization step to compress context. This connects to the context window challenges discussed in <<link "In-Context Learning" "In-Context Learning">><</link>>.

''Orchestration frameworks'' like LangChain, LlamaIndex, and custom pipeline systems provide the scaffolding: they manage prompt templates, handle API calls, parse outputs, implement retry logic, and log intermediate results for debugging. The chain definition is typically declarative — you specify the steps and their connections, and the framework handles execution.

''Retrieval-augmented chains'' integrate <<link "RAG" "RAG">><</link>> at specific steps. For instance, Step 1 generates a search query from the user's question, Step 2 retrieves relevant documents, Step 3 synthesizes an answer from the retrieved context. This is one of the most common prompt chaining patterns in production and is sometimes called a ''RAG pipeline''.

---

!! When Prompt Chaining Breaks Down

''Error propagation across steps:'' Despite the localization benefits, errors in early steps still propagate downstream. If Step 1 extracts the wrong key claims from a document, Steps 2 and 3 will evaluate and summarize //the wrong claims// — producing a polished, confident, completely wrong output. Unlike a monolithic prompt where the error might be visibly incoherent, chained errors can produce outputs that look perfectly reasonable because each individual step executed correctly on its (incorrect) input. Adding validation between steps mitigates this but does not eliminate it — the validator itself can fail to catch subtle errors, especially when the wrong output is plausible.

''Latency accumulation:'' Each step in a chain requires a separate API call, and steps are sequential by definition (except in branching architectures). A 5-step chain with 2-second-per-step latency takes 10 seconds minimum, compared to a single 3-second monolithic call. For interactive applications where users expect sub-second responses, even short chains can be prohibitively slow. Parallelizable branches help but add architectural complexity. Streaming the final step's output can mask some latency, but users still wait for all prior steps to complete before any output appears.

''Over-decomposition and information loss:'' A chain with too many steps loses information at each boundary. Each intermediate output is a lossy compression of the full context available at that step. If Step 2 produces a summary that drops a nuance, Step 3 cannot recover it. The temptation to decompose everything into tiny steps can produce chains where each step is trivially reliable but the end-to-end output is impoverished. The optimal decomposition balances step reliability against information preservation — a problem that currently has no formal solution and must be resolved through empirical testing.

---

!! Technical Details

''Relationship to software engineering:'' Prompt chaining is the LLM analog of the Unix philosophy — each tool does one thing well, and tools compose via pipes. The intermediate outputs are analogous to typed interfaces between microservices. The same software engineering principles apply: keep interfaces narrow and well-defined, fail fast, log everything, and test components independently.

''Relationship to Chain of Thought:'' <<link "Chain of Thought" "Chain of Thought">><</link>> externalizes reasoning within a single prompt. Prompt chaining externalizes it across multiple prompts. The tradeoff: CoT is faster (one API call) but less controllable (the model decides how to reason). Chaining is slower but gives the developer explicit control over each reasoning step, the ability to validate intermediate results, and the option to use different models or configurations for different steps.

''Relationship to agents:'' <<link "LLM Agents" "LLM Agents">><</link>> can be viewed as prompt chains where the model itself decides the next step dynamically, rather than following a pre-defined sequence. Agents are more flexible but harder to debug and less predictable. Many production systems use a hybrid: a fixed chain structure with one or two steps that are internally agentic (the step uses tools and reasoning loops) while the overall flow is deterministic.

''Model mixing:'' A practical advantage of chaining is that different steps can use different models. A cheap, fast model can handle classification or extraction steps, while an expensive, capable model handles only the synthesis step that requires deep reasoning. This can reduce cost by 60-80% compared to routing everything through the most capable model.

''Gate and validation patterns:'' Production chains typically include ''gate'' steps — lightweight checks that verify the intermediate output meets structural requirements (correct JSON schema, expected number of items, no empty fields) before passing it downstream. These gates are often deterministic code, not LLM calls, and they catch the majority of format failures at near-zero cost.

''Prompt chaining vs. fine-tuning:'' For tasks that are highly repetitive and well-defined, fine-tuning a single model to perform the entire task in one pass can be more efficient than chaining. Chaining is preferred when the task is complex, the decomposition is natural, the pipeline needs to be modified frequently, or training data for the end-to-end task is scarce. See <<link "In-Context Learning" "In-Context Learning">><</link>> for how few-shot examples within each chain step can boost per-step reliability without fine-tuning.
