! Layer Normalization

<img src="assets/layer_normalization.svg" alt="Diagram comparing Batch Normalization and Layer Normalization, showing that BN normalizes down columns (across the batch for each feature) while LN normalizes across rows (across features for each sample)" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you are a teacher grading a single student's test. Rather than comparing that student's score on question 5 to every other student's score on question 5 (which is what <<link "Batch Normalization" "Batch Normalization">><</link>> does), you instead look at all of that one student's answers together and ask: "Are these scores on a consistent scale?" If the student scored 95 on easy questions and 12 on hard ones, you re-center and rescale all their answers so the overall profile has a stable mean and spread. That is ''layer normalization'' — it normalizes across the features within a single example, completely ignoring what any other example in the batch looks like.

This batch-independence is what makes layer normalization the default <<link "normalization" "normalization">><</link>> technique for <<link "Transformers" "Transformers">><</link>> and <<link "recurrent neural networks" "RNN">><</link>>. Since each sample is normalized on its own, the method works identically whether your batch size is 1 or 10,000, and it behaves the same way at training and inference time — no running statistics, no train/eval mode switch, no subtle bugs.

---

!! The Core Insight

<<link "Batch Normalization" "Batch Normalization">><</link>> computes the mean and variance of each feature //across the batch//. This works well when batches are large enough that those statistics are reliable estimates of the true distribution. But when the batch is small — or when the data is sequential and each timestep has a different distribution — the batch statistics become noisy and unreliable.

Layer normalization flips the axis entirely. Instead of asking "What is the average value of feature $j$ across all samples?", it asks "What is the average value of //all features// within sample $i$?" For each individual input vector, it computes the mean and variance across the feature dimension, then centers and rescales. The result is that every single sample gets a self-contained normalization, with no dependency on what else happens to be in the batch.

This is why layer normalization became essential for <<link "Transformers" "Transformers">><</link>>. In a Transformer, each token in a sequence is processed with its own feature vector. Layer normalization ensures that each token's representation is well-conditioned, regardless of sequence length, batch size, or whether you are training or running inference. The entire "Attention Is All You Need" architecture — and every large language model built on it — relies on layer normalization at every sublayer.

The key tradeoff: batch normalization has a mild regularizing effect (because the batch statistics introduce noise), while layer normalization does not. In settings where that regularization helps (large-batch CNNs), BN may still be preferred. But in settings where stability and batch-independence matter more, layer normalization wins decisively.

---

!! Pre-Norm vs Post-Norm Placement

In the original Transformer architecture (Vaswani et al., 2017), layer normalization was applied //after// the residual connection — a configuration called ''post-norm'':

$$\text{output} = \text{LayerNorm}(x + \text{Sublayer}(x))$$

This means the sublayer's output is added to the residual, and //then// the sum is normalized. Post-norm Transformers can achieve strong final performance, but they are notoriously difficult to train at depth. The gradients must flow through the normalization layer before reaching the residual path, which can cause instability during the early phase of training. Careful learning rate warmup is typically required.

''Pre-norm'' Transformers flip the order: normalize first, then apply the sublayer, then add the residual:

$$\text{output} = x + \text{Sublayer}(\text{LayerNorm}(x))$$

This arrangement lets gradients flow through the residual connection completely unobstructed — the identity path from output back to input has no normalization layer in the way. Pre-norm architectures train more stably, often without warmup, and scale more predictably to very deep networks. GPT-2, GPT-3, LLaMA, and most modern large language models use pre-norm placement.

The practical takeaway: if you are building a new Transformer, start with pre-norm. Switch to post-norm only if you have evidence that the extra training difficulty is worth the marginal performance gain for your specific task and scale.

---

!! What the Math Is Actually Doing

Given an input vector $\mathbf{x} = (x_1, x_2, \ldots, x_H)$ with $H$ features, layer normalization computes:

$$\mu = \frac{1}{H} \sum_{i=1}^{H} x_i$$

$$\sigma^2 = \frac{1}{H} \sum_{i=1}^{H} (x_i - \mu)^2$$

Then each feature is normalized:

$$\hat{x}_i = \frac{x_i - \mu}{\sqrt{\sigma^2 + \varepsilon}}$$

where $\varepsilon \approx 10^{-5}$ prevents division by zero. Finally, learned parameters $\gamma_i$ and $\beta_i$ (one per feature) apply an affine transformation:

$$y_i = \gamma_i \hat{x}_i + \beta_i$$

The parameters $\gamma$ and $\beta$ are initialized to 1 and 0 respectively, so the network starts with pure normalization and learns to scale and shift each feature independently.

Unlike <<link "Batch Normalization" "Batch Normalization">><</link>>, there are no running statistics to maintain. The mean $\mu$ and variance $\sigma^2$ are computed fresh for each input at both training and inference time. This means layer normalization is a pure, deterministic function of its input — the same input always produces the same output, regardless of what else is in the batch.

For a Transformer with hidden dimension $d = 768$, each layer normalization operation computes a mean and variance over 768 values, applies the normalization, then scales and shifts with $2 \times 768 = 1536$ learned parameters. Across a 12-layer Transformer with two layer norms per block (one for attention, one for the feed-forward network), that is $12 \times 2 \times 1536 = 36{,}864$ parameters dedicated to normalization — a tiny fraction of the total model size.

---

!! Practical Considerations

''RMSNorm — a faster variant.'' RMSNorm (Root Mean Square Normalization) drops the mean-centering step entirely and normalizes by the root mean square alone:

$$\hat{x}_i = \frac{x_i}{\sqrt{\frac{1}{H}\sum_{j=1}^{H} x_j^2 + \varepsilon}}$$

This saves one pass over the data (no need to compute $\mu$) and has been shown to match layer normalization's performance in practice. LLaMA, Gemma, and several other recent architectures use RMSNorm instead of standard layer normalization. When choosing between them, RMSNorm is the pragmatic default unless you have evidence that mean-centering matters for your specific task.

''Numerical precision matters.'' The normalization computation involves subtracting the mean from each value and dividing by a potentially small standard deviation. In half-precision (FP16), this can cause numerical instability — the subtraction can lose significant bits when values are close to the mean. Most frameworks compute layer normalization internally in FP32 even when the rest of the network runs in FP16 or BF16. If you implement a custom layer norm kernel, always accumulate the mean and variance in FP32.

''Interaction with <<link "gradient descent" "Stochastic Gradient Descent">><</link>>.'' Layer normalization smooths the loss landscape by reducing the sensitivity of each layer's output to the scale of its input. If the pre-normalization activations are scaled by a constant $c$, the post-normalization values are unchanged (the mean and variance absorb the scaling). This scale invariance means the effective learning rate is automatically adjusted for each layer, which is one reason why layer-normalized networks are less sensitive to learning rate choices.

---

!! When Layer Normalization Breaks Down

''1. Convolutional networks with spatial structure.'' Layer normalization computes statistics across //all// features in a sample, which in a CNN means pooling across both spatial positions and channels. This conflates spatial information with channel information, destroying the channel-wise structure that convolutional layers depend on. For CNNs, <<link "Batch Normalization" "Batch Normalization">><</link>> (or Group Normalization for small batches) remains the better choice because it normalizes each channel independently across the batch and spatial dimensions, preserving the channel structure.

''2. Features with deliberately different scales.'' Layer normalization assumes that all features in a vector should be on a roughly comparable scale. If your architecture intentionally encodes information in the //relative magnitudes// of different features — for example, if feature 1 is always 100x larger than feature 2 and that ratio carries meaning — then layer normalization will erase that signal by forcing all features to the same scale. The learned $\gamma$ and $\beta$ can partially recover this, but only after training discovers the pattern. In such cases, consider normalizing over subsets of features (group normalization) rather than all features at once.

''3. Very low-dimensional feature spaces.'' When the feature dimension $H$ is very small (say, $H = 4$ or $H = 8$), the mean and variance computed over so few values are noisy estimates. The normalization step can amplify small perturbations rather than stabilizing the representation. This is rarely a problem in modern <<link "Transformers" "Transformers">><</link>> (where $H \geq 256$), but it can arise in lightweight models, early layers of hierarchical architectures, or when layer normalization is applied to small embedding dimensions. If you see training instability with very small feature dimensions, consider removing the normalization or increasing the dimension.

---

!! Technical Details

''Original paper:'' Ba, Kiros & Hinton, 2016 — "Layer Normalization." Proposed as a solution to BN's batch-size dependency, specifically motivated by <<link "RNN" "RNN">><</link>> training where BN was impractical.

''Parameter count:'' $2H$ per layer norm operation ($H$ for $\gamma$, $H$ for $\beta$). For a Transformer with hidden size $d$, each block typically has 2 layer norms, contributing $4d$ parameters per block.

''Computational cost:'' Two passes over the feature vector per normalization (one for $\mu$, one for $\sigma^2$), plus elementwise scaling. Total: $O(H)$ per sample per layer norm. In practice, fused CUDA kernels compute mean and variance in a single pass using Welford's online algorithm.

''Gradient:'' The Jacobian of layer normalization is not diagonal — normalizing $x_i$ depends on all $x_j$ through the shared $\mu$ and $\sigma^2$. The backward pass must account for the contribution of each $x_j$ to the mean and variance. Efficient implementations compute this analytically rather than through autograd, reducing memory usage.

''Variants:''
- ''RMSNorm:'' Drops mean centering. Used in LLaMA, PaLM, Gemma.
- ''AdaLN (Adaptive Layer Norm):'' Conditions $\gamma$ and $\beta$ on an external signal (e.g., timestep in diffusion models). Used in DiT and Stable Diffusion 3.
- ''QK-Norm:'' Applies layer norm to query and key vectors before attention dot products. Stabilizes training at large scale.

''Typical placement in Transformers:''
- ''Post-norm (original):'' $\text{LN}(x + \text{Attn}(x))$ — harder to train, potentially better final quality.
- ''Pre-norm (modern default):'' $x + \text{Attn}(\text{LN}(x))$ — easier to train, scales reliably.
- ''Sandwich norm:'' Applies LN both before and after the sublayer. Occasionally used for extra stability.

''Framework defaults:'' `torch.nn.LayerNorm(normalized_shape)` in PyTorch; `tf.keras.layers.LayerNormalization(axis=-1)` in TensorFlow/Keras. Both default to normalizing over the last dimension (features).
