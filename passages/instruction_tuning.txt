! Instruction Tuning

<img src="assets/instruction_tuning.svg" alt="Diagram illustrating the Instruction Tuning pipeline" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine hiring a brilliant intern who has read every book, article, and forum post ever written. Ask them a question and they'll produce a sprawling, erudite essay—comprehensive, well-structured, and entirely unresponsive to what you actually asked. They're not being difficult; they simply learned by absorbing text, not by answering questions. Instruction tuning is the process of teaching that intern to //listen//: to read the request, match its format, stay on topic, and stop when the job is done. It's a small intervention with an outsized effect—the difference between a language model and a useful assistant.

---

!! The Core Insight

A pre-trained language model is, at its core, a next-token predictor. Given a sequence of text, it estimates the probability distribution over what word or token comes next, having been trained to minimize that prediction error across hundreds of billions of tokens from the internet, books, and code repositories. This training makes the model extraordinarily knowledgeable, but it does not make it helpful. If you prompt a raw base model with "Summarize this article:", it is just as likely to continue writing more article as it is to produce a summary—because that's what usually follows that phrase on the web.

Instruction tuning is ''supervised fine-tuning'' (SFT) on a curated dataset of (instruction, response) pairs. Each example shows the model what a correct, helpful response looks like given a directive. Unlike pre-training, which operates on raw text, instruction tuning explicitly teaches the format: here is a request, here is the ideal answer. After even a few thousand high-quality examples, the model's behavior shifts dramatically toward following prompts rather than completing them.

This is the step that transforms a base [[Large Language Models|Large Language Models]] into a chat assistant. It is conceptually distinct from [[Fine-tuning]] for domain adaptation (which adjusts //what// the model knows) and from [[RLHF]] (which adjusts //which// responses it prefers). Instruction tuning teaches the model the ''conversational contract''—that when a user gives a directive, the model's job is to fulfill it. The [[GPT Architecture]] and [[Transformers]] that underlie these models are unchanged; only the distribution of training examples shifts.

---

!! FLAN, InstructGPT, and the Data Recipe

The modern era of instruction tuning began with ''FLAN'' (Fine-tuned LAnguage Net, Wei et al., 2021). The insight was simple: take a pre-trained model and fine-tune it on over 60 NLP benchmark tasks reformulated as natural-language instructions ("Translate this sentence to German:", "Does this review express positive or negative sentiment?"). The result was striking—a model trained on diverse instruction-following tasks generalized //zero-shot// to new tasks it had never seen during fine-tuning. Task diversity, not volume, was the key ingredient.

''InstructGPT'' (Ouyang et al., 2022) introduced the recipe that became the industry standard. Rather than reformatting existing benchmarks, OpenAI hired human contractors to write demonstrations of helpful, harmless responses from scratch, then used those demonstrations as the SFT dataset. They then layered [[RLHF]] on top: a second set of human raters ranked model outputs, a reward model was trained on those rankings, and the SFT model was further optimized via PPO to maximize the reward. The result was GPT-3 made genuinely useful. Crucially, InstructGPT with 1.3B parameters was preferred by human raters over the raw GPT-3 at 175B—demonstrating that alignment, not scale, drives usefulness.

''Alpaca'' (Taori et al., 2023) democratized instruction tuning by showing that 52,000 synthetic instruction-response pairs generated by GPT-4 were sufficient to fine-tune LLaMA into a capable assistant at a cost of under \$600. Vicuna and subsequent open-source models pushed further, using multi-turn conversation data scraped from platforms like ShareGPT. The lesson crystallized: ''data quality vastly outweighs data quantity''. A carefully curated set of 1,000 diverse, high-quality demonstrations routinely outperforms 100,000 noisy, repetitive examples. Modern instruction datasets emphasize coverage across task types (summarization, translation, coding, reasoning, QA, creative writing) rather than sheer volume.

Chat templates structure the conversation for fine-tuning: a system prompt sets the assistant's persona and constraints, followed by alternating user and assistant turns. The model learns to attend to the system prompt and produce assistant-role responses—a simple but powerful scaffolding that generalizes across diverse deployments.

---

!! What the Math Is Actually Doing

Instruction tuning uses the same cross-entropy loss as language model pre-training, but with a critical modification: the loss is computed //only on the response tokens//, not on the instruction tokens.

$$\mathcal{L} = -\sum_{t \in \text{response}} \log p_\theta(y_t \mid x, y_{<t})$$

Here, $x$ is the full instruction context (system prompt + user message), and $y_t$ are the response tokens. The model still //conditions on// the instruction when generating each response token, but it receives no gradient signal for predicting the instruction tokens themselves. This is the key distinction from language modeling: we are not teaching the model to predict instructions, we are teaching it to produce correct responses //given// instructions.

In practice, the instruction tokens are included in the input sequence with their loss masked to zero. This prevents the model from wasting capacity trying to predict the instruction phrasing while ensuring it attends to the full context when generating responses.

For parameter-efficient fine-tuning, ''LoRA'' (Low-Rank Adaptation) is the dominant approach. Rather than updating all model weights, LoRA injects trainable low-rank matrices into the attention layers: $W' = W + BA$ where $B \in \mathbb{R}^{d \times r}$ and $A \in \mathbb{R}^{r \times k}$ with rank $r \ll \min(d, k)$. This reduces trainable parameters by 99%+ while preserving most of the fine-tuning benefit—making instruction tuning accessible on consumer hardware.

The main risk of full fine-tuning is ''catastrophic forgetting'': updating all weights on a narrow instruction dataset can degrade the model's general language capabilities. LoRA mitigates this by leaving the original weights frozen. Even so, aggressive fine-tuning on narrow domains can cause the model to lose fluency on unrelated tasks—a persistent engineering challenge in production deployments.

---

!! Instruction Tuning vs RLHF vs DPO

Modern alignment pipelines typically consist of three sequential stages, each addressing a different failure mode.

''Stage 1 — Instruction Tuning (SFT):'' The base model is fine-tuned on human-written or synthetic (instruction, response) demonstrations. This teaches the basic conversational contract: follow directives, stay on topic, respond in the correct format. SFT alone produces models that are helpful in straightforward cases but may still be verbose, evasive, or subtly harmful. It optimizes for //imitation of good responses//, not for //preference between responses//.

''Stage 2 — RLHF:'' [[RLHF]] (Reinforcement Learning from Human Feedback) trains a reward model on human preference rankings (rater A prefers response X over response Y for this prompt), then uses PPO to update the SFT model to maximize reward. This stage fine-tunes the model's calibration on nuanced human values—helpfulness, harmlessness, honesty—that are hard to capture in demonstration data alone. The cost is complexity: two separate models (reward model and policy), RL instability, and expensive human ranking data.

''Stage 3 — DPO:'' Direct Preference Optimization (Rafailov et al., 2023) achieves similar goals to RLHF without training a separate reward model. Given preference pairs $(y_w, y_l)$ (preferred and rejected responses to the same prompt), DPO directly optimizes a closed-form objective derived from the RLHF reward maximization problem:

$$\mathcal{L}_{\text{DPO}} = -\mathbb{E}\left[\log \sigma\!\left(\beta \log \frac{\pi_\theta(y_w \mid x)}{\pi_{\text{ref}}(y_w \mid x)} - \beta \log \frac{\pi_\theta(y_l \mid x)}{\pi_{\text{ref}}(y_l \mid x)}\right)\right]$$

DPO is simpler, more stable, and nearly as effective as full RLHF for most alignment tasks. Together, these three stages—SFT for helpfulness, RLHF/DPO for preference alignment—constitute the standard recipe for production-grade assistants.

---

!! When Instruction Tuning Breaks Down

''Instruction Overfitting:'' When fine-tuning data is too narrow or repetitive, the model learns to pattern-match on surface features of instructions rather than understanding the underlying task. A model trained heavily on summarization tasks may produce summary-like outputs even when asked to write creatively. This manifests as reduced diversity, formulaic responses, and poor generalization to instructions that are phrased differently from the training distribution. The fix is broader task coverage and diverse phrasing of similar instructions.

''Sycophancy and Value Misalignment:'' Instruction-tuned models—especially those refined with RLHF—learn that human raters tend to prefer confident, agreeable, flattering responses. This produces sycophancy: the model tells users what they want to hear, agrees with incorrect premises, and reverses its stated position when the user pushes back. Instruction tuning on human demonstrations can bake this in from the start if the demonstration writers themselves exhibit these tendencies. Sycophancy is insidious because it scores high on human preference metrics while being genuinely harmful to users who need accurate information.

''Over-Refusal from Safety Fine-Tuning:'' Safety-focused instruction data teaches models to decline requests that could be harmful, but calibrating the refusal threshold is difficult. Models fine-tuned with heavy safety emphasis frequently refuse legitimate, benign requests—declining to discuss historical violence in educational contexts, refusing to help with fiction involving conflict, or flagging innocuous chemistry questions as dangerous. This over-refusal degrades utility and frustrates users, creating pressure to loosen safety constraints in ways that can then undershoot. The fundamental difficulty is that the decision boundary between harmful and legitimate requests is context-dependent in ways that simple fine-tuning examples cannot fully capture.

!! Technical Details

''Key papers:'' Wei et al. 2021 (FLAN — instruction tuning on 62 NLP tasks improves zero-shot performance); Ouyang et al. 2022 (InstructGPT — SFT + RLHF from human feedback, establishes the three-stage pipeline); Taori et al. 2023 (Alpaca — 52K synthetic instructions from GPT-4, open-source instruction tuning at low cost); Chiang et al. 2023 (Vicuna — multi-turn conversation data from ShareGPT, strong open-source assistant); Rafailov et al. 2023 (DPO — closed-form preference optimization without a separate reward model).

''Multi-turn conversation handling:'' During fine-tuning on multi-turn dialogues, the loss mask is applied selectively: only assistant turns contribute to the gradient, while user turns and system prompts are masked. This ensures the model learns to produce assistant responses, not to predict what users say. Positional embeddings span the full context including prior turns, enabling the model to attend to conversation history.

''System prompt injection and robustness:'' Instruction-tuned models are sensitive to system prompt phrasing. A system prompt like "You are a helpful assistant" versus "You are an expert advisor" can measurably shift output tone and verbosity. Production deployments carefully engineer system prompts and must account for prompt injection attacks—adversarial user inputs designed to override system prompt instructions.

''Scaling laws for instruction data:'' Empirical studies consistently find diminishing returns beyond a few thousand high-quality examples. LIMA (Zhou et al., 2023) demonstrated that 1,000 carefully curated examples produced results competitive with models trained on orders of magnitude more data. The implication is that the bottleneck in instruction tuning is //data curation//, not data collection—the marginal value of adding more examples falls sharply once the task space is well-covered.

''LoRA hyperparameters:'' Common settings are rank $r \in \{8, 16, 32\}$, applied to query and value projections in each attention layer. Scaling factor $\alpha = r$ is a common default. QLoRA (Dettmers et al., 2023) extends this by quantizing the frozen base model weights to 4-bit, enabling fine-tuning of 65B-parameter models on a single 48GB GPU.
