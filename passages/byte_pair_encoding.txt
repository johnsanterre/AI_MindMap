! Byte Pair Encoding
<img src="assets/byte_pair_encoding.svg" alt="Byte Pair Encoding merge process diagram" style="max-width:100%; margin:0.5em 0 1em 0;">

Dictionaries compress language by listing whole words — efficient for common words, but helpless against new ones. Morse code uses single characters — handles everything but is verbose. ''Byte Pair Encoding'' finds the sweet spot: a vocabulary of subword units that handles both common words efficiently and rare or unknown words gracefully. It is the tokenization algorithm underlying GPT-2, GPT-3/4, LLaMA, RoBERTa, and most modern large language models.

!! The Core Insight

Sennrich et al. (2016) adapted BPE — originally a lossless data-compression algorithm — for NLP tokenization. The idea is disarmingly simple: start with a vocabulary of individual characters, then ''iteratively merge the most frequent adjacent pair of tokens'' into a single new token, repeating until the vocabulary reaches a target size (typically 30 K–100 K). The result is a vocabulary that naturally allocates tokens to frequency: common words collapse to single tokens; rare words split into recognizable subword pieces; genuinely unknown words degrade gracefully to character sequences rather than a single UNK symbol.

Links: [[Tokenizer]], [[Large Language Models]], [[Bert]], [[GPT Architecture]], [[Word Embeddings]]

!! The BPE Algorithm

Initialization: vocabulary = all Unicode characters (or raw bytes for byte-level BPE) plus any special tokens (e.g., &lt;|endoftext|&gt;).

Repeat K times:
# Count every adjacent pair of tokens across the training corpus.
# Select the pair (a, b) with the highest count.
# Create a new token ab, add it to the vocabulary.
# Replace every occurrence of the sequence a b in the corpus with ab.

Encoding at inference time: apply the learned merge rules in the same order they were created — earliest merges first — greedy left-to-right. The merge order is deterministic and must be stored alongside the vocabulary file.

//Example:// "lower" tokenizes to ["low", "er"] if "low" was a learned merge but "lower" never reached merge threshold.

!! What the Math Is Actually Doing

BPE greedily minimizes a proxy for total description length:

$$\text{cost} = |\text{vocabulary}| + |\text{encoded corpus}|$$

Each merge step reduces corpus length by exactly $f(a, b)$ (the frequency of the merged pair) while increasing vocabulary size by one. The algorithm accepts a merge whenever the corpus-length reduction outweighs the vocabulary-size cost. This is a greedy heuristic — not globally optimal compression — but it runs in near-linear time and produces excellent subword units in practice.

''SentencePiece'' (Kudo & Richardson, 2018) reimplements BPE and unigram-LM tokenization directly on raw Unicode without any pre-tokenization step, making it fully language-agnostic. ''Unigram LM tokenization'' is a probabilistic alternative: it builds a vocabulary by starting large and pruning tokens that contribute least to corpus likelihood, and assigns probabilities to multiple possible segmentations of a string — useful for training with subword regularization.

!! Vocabulary Size Tradeoffs

||Vocabulary size||Effect||
|Small (≤ 2 K)|Long sequences; model must learn to compose meaning from characters; computationally expensive.|
|Medium (32 K–50 K)|Typical for monolingual English models (GPT-2: 50 K, LLaMA: 32 K). Good balance of sequence length and coverage.|
|Large (100 K+)|Fewer tokens per word for common vocabulary; embedding matrix becomes large; rare subwords receive few training examples. GPT-4 uses cl100k_base (~100 K).|
|Multilingual (≥ 50 K)|Necessary to give non-Latin scripts reasonable coverage; otherwise Arabic, Chinese, or Thai text requires 3–4× more tokens per English equivalent — the //"tokenization tax."//|

!! When BPE Breaks Down

''Token boundary effects:'' because whitespace is folded into tokens, "dog" and " dog" are distinct tokens with separate embedding rows. This causes subtle inconsistencies when a word appears at the start vs. middle of a sentence.

''Arithmetic and structured data:'' numbers are split at arbitrary boundaries ("1,234" → ["1", ",", "234"] or ["12", "34"]) with no principled alignment to numerical magnitude. Models trained on BPE tokens struggle with digit-level arithmetic.

''Non-Latin scripts:'' byte-level BPE (used in GPT-2/3/4) encodes every Unicode character via its UTF-8 bytes, guaranteeing no UNK tokens but yielding very long token sequences for scripts outside the training distribution — degrading both quality and cost for those languages.

!! Technical Details

''Key papers:'' Sennrich, Haddow & Birch (2016) — BPE for neural machine translation; Kudo & Richardson (2018) — SentencePiece; Kudo (2018) — Unigram LM subword regularization; Radford et al. (2019) — GPT-2 byte-level BPE.

''Libraries:'' tiktoken (OpenAI, Rust-backed, fast), HuggingFace tokenizers (Rust core, Python bindings), SentencePiece (Google, C++).

''WordPiece'' (used in BERT): superficially similar to BPE but selects merges by likelihood gain rather than raw frequency — tends to prefer linguistically meaningful splits.

''Tokenization-free alternatives:'' ByT5 operates directly on UTF-8 bytes; character-level language models (e.g., CharFormer) process individual characters but require significantly more compute per token.

Typical merge counts: GPT-2 performs 50 K merges; LLaMA tokenizer performs ~32 K merges on a SentencePiece BPE model trained on 2 T tokens of multilingual text.
