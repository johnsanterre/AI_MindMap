! Group Query Attention

<img src="assets/group_query_attention.svg" alt="Diagram comparing Multi-Head Attention, Multi-Query Attention, and Group Query Attention architectures" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine an orchestra where each musician has their own personal music stand holding their sheet music. That works, but it takes up a lot of space and a lot of stands. Now notice that the violins are all playing the same part — so four violinists can share one stand without missing a note. ''Group Query Attention'' (GQA) applies exactly this logic to the Transformer attention mechanism. In standard <<link "Multi-Head Attention" "Multi-Head Attention">><</link>>, every query head maintains its own dedicated key and value projections, and the memory cost of caching those keys and values grows linearly with the number of heads. GQA groups query heads together and lets each group share a single set of key-value heads, cutting the <<link "KV Cache" "KV Cache">><</link>> memory by 4-8× with minimal quality loss. The queries themselves remain independent — each musician still reads the notes with their own eyes — only the stands are shared.

!! The Core Insight

There are three points on the spectrum of key-value sharing in <<link "Attention" "Attention">><</link>>:

''Multi-Head Attention (MHA):'' Every head $h$ has its own $Q_h$, $K_h$, $V_h$. If you have $H = 32$ heads, you store 32 sets of keys and values in the cache. This is maximally expressive but maximally expensive in memory.

''Multi-Query Attention (MQA):'' //All// heads share a single $K$ and $V$. Maximum compression — the cache shrinks by a factor of $H$ — but quality can drop noticeably because every head is forced to attend over identical keys and values, losing the diversity that makes multi-head attention powerful.

''Group Query Attention (GQA):'' The middle ground. You define $G$ groups of query heads, and each group shares one set of $K, V$ heads. When $G = H$, you recover MHA (no sharing). When $G = 1$, you recover MQA (full sharing). The sweet spot is typically $G = H/4$ or $G = H/8$ — enough sharing to slash memory, enough diversity to preserve quality.

!! The Memory Math

The <<link "KV Cache" "KV Cache">><</link>> stores keys and values for all previously generated tokens so they don't need to be recomputed. The memory cost per layer is:

$$\text{KV cache per layer} = 2 \times n_{\text{kv\_heads}} \times d_{\text{head}} \times L \times \text{bytes per param}$$

where $L$ is the sequence length and the factor of 2 accounts for both keys and values.

Consider a model with 64 query heads and $d_{\text{head}} = 128$ using FP16 (2 bytes):

* ''MHA'' (64 KV heads): $2 \times 64 \times 128 \times L \times 2 = 32{,}768 \cdot L$ bytes per layer
* ''GQA with 8 groups'' (8 KV heads): $2 \times 8 \times 128 \times L \times 2 = 4{,}096 \cdot L$ bytes per layer — an ''8× reduction''
* ''MQA'' (1 KV head): $2 \times 1 \times 128 \times L \times 2 = 512 \cdot L$ bytes per layer — a 64× reduction but at a quality cost

''Llama 2 70B'' uses GQA with 8 KV heads serving 64 query heads across 80 layers. At a sequence length of 4096 tokens in FP16, the KV cache drops from ~40 GB (MHA) to ~5 GB (GQA). This is the difference between fitting on one GPU and needing several.

!! What the Math Is Actually Doing

In standard Multi-Head Attention, each head $h \in \{1, \ldots, H\}$ computes its own query, key, and value projections:

$$Q_h = X W_h^Q, \quad K_h = X W_h^K, \quad V_h = X W_h^V$$

where $Q_h, K_h, V_h \in \mathbb{R}^{L \times d_h}$ and attention is:

$$\text{head}_h = \text{softmax}\!\left(\frac{Q_h K_h^\top}{\sqrt{d_h}}\right) V_h$$

In GQA, we define a mapping function $g(h)$ that assigns each query head to a KV group:

$$g(h) = \left\lfloor \frac{h \cdot G}{H} \right\rfloor$$

Each query head $h$ keeps its own $Q_h = X W_h^Q$ but shares key-value projections within its group:

$$\text{head}_h = \text{softmax}\!\left(\frac{Q_h K_{g(h)}^\top}{\sqrt{d_h}}\right) V_{g(h)}$$

The queries remain fully independent — this preserves the model's ability to attend to different aspects of the input from different heads. Only the keys and values are shared, which is equivalent to //upsampling//: you repeat each $K_g, V_g$ a total of $H/G$ times before computing attention. The gradient still flows back through each group's shared $W_g^K$ and $W_g^V$, aggregating information from all query heads in that group.

!! GQA in Practice

Several major <<link "Large Language Models" "Large Language Models">><</link>> use GQA as their default attention mechanism:

* ''Llama 2 70B:'' 64 query heads, 8 KV heads ($G = 8$, ratio 8:1). The 7B and 13B variants use standard MHA; GQA was adopted specifically for the 70B model where KV cache memory becomes a bottleneck.
* ''Mistral 7B:'' 32 query heads, 8 KV heads ($G = 8$, ratio 4:1). Uses GQA even at 7B scale, combined with <<link "Sliding Window Attention" "Sliding Window Attention">><</link>> for further efficiency.
* ''Gemma:'' Various configurations across model sizes, all using GQA.
* ''Llama 3:'' Continued the GQA design across all model sizes.

''Uptraining from MHA:'' You don't have to train a GQA model from scratch. Ainslie et al. showed you can start with an MHA-pretrained model, mean-pool the KV projection weights within each group (average the $W^K$ and $W^V$ matrices of heads assigned to the same group), then fine-tune briefly. This recovers near-original quality with a fraction of the cache cost — typically 5-10% of the original pretraining compute suffices.

!! When GQA Breaks Down

''Too few KV groups destroy quality on diverse-attention tasks.'' When a model needs genuinely different key-value representations across heads — for example, some heads tracking syntactic structure while others track semantic similarity — forcing them to share keys and values loses critical information. Pushing $G$ too low (close to MQA) on tasks requiring this diversity causes measurable degradation, particularly on long-context reasoning and multi-hop retrieval tasks.

''Uptraining doesn't fully recover from-scratch quality.'' Models trained with GQA from the start slightly outperform models that were first trained with MHA and then converted via mean-pooling and fine-tuning. The mean-pooling initialization is a reasonable approximation but imperfect — heads within a group may have learned substantially different key-value representations that averaging destroys. The gap is small (typically <1% on benchmarks) but consistent.

''GQA reduces memory, not compute.'' The number of floating-point operations for attention is unchanged compared to MHA because every query head still computes full attention — $Q_h K_{g(h)}^\top$ is the same shape regardless of whether $K_{g(h)}$ is shared. GQA's benefit is purely in memory bandwidth and cache size, not in FLOPs. For compute-bound scenarios (short sequences, large batch sizes), GQA offers little advantage. It shines in memory-bound scenarios: long sequences, autoregressive decoding, and limited GPU memory.

!! Technical Details

''Origins:'' Multi-Query Attention was introduced by Shazeer (2019) in "Fast Transformer Decoding: One Write-Head is All You Need." Group Query Attention was proposed by Ainslie et al. (2023) in "GQA: Training Generalized Multi-Query Transformer Models from Multi-Head Checkpoints," which also introduced the uptraining procedure.

''Relationship to other techniques:'' GQA is orthogonal to and compatible with <<link "Flash Attention" "Flash Attention">><</link>> (which optimizes attention //computation//), <<link "Sliding Window Attention" "Sliding Window Attention">><</link>> (which limits attention //range//), and quantized KV caches (which reduce //precision//). Combining all four yields multiplicative memory savings.

''Implementation detail:'' In practice, GQA is implemented by broadcasting (repeating) each KV head $H/G$ times along the head dimension before computing attention, making it a drop-in replacement for MHA with no changes to the attention kernel itself.

See also: <<link "Transformers" "Transformers">><</link>>, <<link "Attention" "Attention">><</link>>, <<link "Multi-Head Attention" "Multi-Head Attention">><</link>>, <<link "KV Cache" "KV Cache">><</link>>, <<link "Flash Attention" "Flash Attention">><</link>>, <<link "Large Language Models" "Large Language Models">><</link>>, <<link "Sliding Window Attention" "Sliding Window Attention">><</link>>.
