! Hashing

<img src="assets/hashing.svg" alt="Hashing" width="800">

Imagine you run a library with a million books but no catalog. Every time someone asks for a title, you walk every aisle until you find it — agonizingly slow. Now imagine you invent a rule: take the first letter of the title, convert it to a number, and that number tells you exactly which shelf to check. You have just invented a crude ''hash function''. Real hash functions are far more sophisticated, but the principle is the same: transform an input of arbitrary size into a fixed-size address so you can jump straight to the right location instead of searching. Hashing is one of the most fundamental ideas in computer science, underpinning everything from Python dictionaries and database indexes to [[RSA Cryptography|RSA Cryptography]] and blockchain verification.

!! The Core Insight

A hash function takes any input — a string, a file, a number of any size — and produces a fixed-length output, called the ''hash value'' or ''digest''. The magic lies in three properties. First, it is ''deterministic'': the same input always yields the same output. Second, it is ''efficiently computable'': producing the hash takes time proportional to the input length, not the size of the output space. Third, a good hash function distributes outputs ''uniformly'' across the available range, minimizing the chance that two different inputs land in the same slot.

This uniformity is what makes hash-based data structures fast. A ''hash table'' uses the hash value as an index into an array of buckets. Instead of searching through $n$ items one by one ($O(n)$ time), you compute a hash, jump to the right bucket, and find your item in $O(1)$ average time. The price you pay is that two different keys can map to the same bucket — a ''collision'' — and you need a strategy to handle it.

!! Hash Functions and Collision Resolution

The simplest hash function is ''division hashing'': $h(k) = k \bmod m$, where $m$ is the table size. This works, but the choice of $m$ matters enormously. If $m$ is a power of 2, the hash only depends on the lowest-order bits of $k$, creating patterns. Choosing $m$ as a prime not close to any power of 2 helps distribute keys more evenly — a practical consequence of [[Modular Arithmetic]].

''Multiplication hashing'' avoids the sensitivity to $m$ by multiplying $k$ by an irrational constant (Knuth recommends $A = (\sqrt{5} - 1)/2 \approx 0.618$), extracting the fractional part, and scaling: $h(k) = \lfloor m \cdot (kA \bmod 1) \rfloor$. This spreads keys more uniformly regardless of $m$.

When collisions occur, there are two main strategies. ''Chaining'' stores all items that hash to the same bucket in a linked list (or tree). Lookup walks the chain — $O(1)$ on average if the load factor $\alpha = n/m$ stays low. ''Open addressing'' finds an alternative slot within the table itself. ''Linear probing'' checks the next slot, then the next, and so on. ''Quadratic probing'' uses a quadratic sequence of offsets. ''Double hashing'' uses a second hash function to determine the probe step. Open addressing avoids pointer overhead but degrades sharply as the table fills.

!! What the Math Is Actually Doing

A hash function $h: U \to \{0, 1, \dots, m-1\}$ maps a large universe $U$ of possible keys to $m$ table slots. Since $|U| \gg m$, the [[Pigeonhole Principle]] guarantees that collisions are inevitable — multiple keys //must// share the same hash value.

Under the ''simple uniform hashing assumption'' (SUHA), each key is equally likely to hash to any of the $m$ slots, independently. The expected number of keys per bucket is $\alpha = n/m$. For chaining, the expected time for a successful search is:

$$T_{\text{success}} = 1 + \frac{\alpha}{2}$$

and for an unsuccessful search:

$$T_{\text{fail}} = 1 + \alpha$$

Both are $O(1)$ when $\alpha$ is bounded by a constant — which you ensure by resizing the table when $\alpha$ exceeds a threshold.

For open addressing with linear probing, Knuth's analysis gives the expected number of probes for an unsuccessful search as:

$$E[\text{probes}] \approx \frac{1}{2}\left(1 + \frac{1}{(1-\alpha)^2}\right)$$

This blows up as $\alpha \to 1$: at 90% full, you expect roughly 50 probes. This is why open-addressing tables resize aggressively.

''Cryptographic hash functions'' (SHA-256, BLAKE3) add stronger requirements: ''preimage resistance'' (given $h(x)$, finding $x$ is computationally infeasible), ''second preimage resistance'' (given $x$, finding $x' \neq x$ with $h(x') = h(x)$ is infeasible), and ''collision resistance'' (finding //any// pair with the same hash is infeasible). The birthday paradox from [[Pigeonhole Principle]] combinatorics means that for a hash with $b$-bit output, a collision can be found in roughly $O(2^{b/2})$ trials — which is why SHA-256's 256-bit output requires $\sim 2^{128}$ work to break, placing it far beyond brute-force reach.

!! Hashing in Machine Learning and Practice

''Feature hashing'' (the "hashing trick") is widely used in ML when the feature space is enormous. Instead of maintaining a vocabulary mapping every token to a column index, you hash each feature name to an index in a fixed-size vector: $\phi_j(x) = \sum_{i: h(f_i) = j} \xi(f_i) \cdot x_i$, where $\xi$ is a sign hash that helps cancel collision bias in expectation. This technique powers large-scale text classification (Vowpal Wabbit) and recommendation systems where feature vocabularies reach billions.

''Locality-Sensitive Hashing'' (LSH) deliberately violates the uniform-distribution goal. It designs hash functions so that similar items are //more likely// to collide. For cosine similarity, random hyperplane hashing projects vectors onto random directions and records the sign. Two vectors that are close in angle will agree on most hyperplane signs, landing in the same bucket. LSH powers approximate nearest neighbor search in high-dimensional spaces — critical for retrieval-augmented generation, image search, and deduplication.

''Bloom filters'' use multiple hash functions to compactly represent a set. Insert an element by setting bits at $k$ hash positions; query by checking whether all $k$ bits are set. False positives are possible (all bits set by coincidence) but false negatives are not. With $m$ bits, $n$ elements, and $k = (m/n) \ln 2$ hash functions, the false positive rate is approximately $(1 - e^{-kn/m})^k$. Bloom filters are used in database systems, network routers, and spell checkers where memory is scarce.

!! When Hashing Breaks Down

* ''Adversarial collisions and hash flooding.'' If an attacker knows your hash function, they can craft inputs that all hash to the same bucket, turning $O(1)$ lookups into $O(n)$ — a denial-of-service attack. Web frameworks have been hit by this: an attacker sends thousands of POST parameters designed to collide. The fix is ''universal hashing'' or ''SipHash'' — keyed hash functions where the key is randomized at startup, so the attacker cannot predict collisions.

* ''Poor load factor management.'' When a hash table grows too full without resizing, performance collapses. For open addressing, crossing $\alpha = 0.75$ begins a steep degradation curve. For chaining, long chains defeat the purpose of hashing. Amortized doubling (resize when $\alpha$ exceeds a threshold, rehashing all entries) maintains $O(1)$ average time but introduces occasional $O(n)$ resize pauses — problematic in real-time systems. Consistent hashing and incremental resizing address this for distributed systems.

* ''Hash function weakness in cryptographic contexts.'' Using a non-cryptographic hash (CRC32, MurmurHash) where security is needed is dangerous — these functions are fast but trivially invertible. MD5 and SHA-1, once considered secure, now have known collision attacks. Choosing the wrong hash function for the context — speed-optimized for hash tables vs. security-hardened for digital signatures and password storage — can lead to catastrophic vulnerabilities.

!! Technical Details

''Universal hashing'' selects $h$ randomly from a family $\mathcal{H}$ such that for any two distinct keys $x \neq y$, $\Pr_{h \in \mathcal{H}}[h(x) = h(y)] \leq 1/m$. The Carter-Wegman family $h_{a,b}(x) = ((ax + b) \bmod p) \bmod m$ (where $p$ is prime, $a \neq 0$) achieves this and is fast to compute.

''Perfect hashing'' achieves $O(1)$ worst-case lookup for static key sets. The FKS (Fredman-Komlós-Szemerédi) scheme uses two levels: the first level hashes into $m$ buckets, and each bucket uses a second-level hash table sized as the square of the number of keys in that bucket. Total space remains $O(n)$.

''Consistent hashing'' (Karger et al., 1997) maps both keys and servers to a ring using a hash function. Each key is assigned to the nearest server clockwise. When a server is added or removed, only $O(n/m)$ keys need to be remapped — critical for distributed caches (Memcached, Amazon DynamoDB).

''Cuckoo hashing'' uses two hash functions and two tables. Each key has exactly two candidate positions. On collision, the existing key is "kicked out" to its alternative position, potentially triggering a chain of evictions. Lookup is always $O(1)$ worst-case (check two positions). Insertion is $O(1)$ amortized but can require table rebuilding if eviction cycles occur.

''Robin Hood hashing'' is a variant of open addressing where, during insertion, a new key can displace an existing key that is closer to its ideal position. This equalizes probe distances across all keys, dramatically reducing variance and worst-case lookup time.

Password storage uses ''slow'' hash functions by design. ''bcrypt'', ''scrypt'', and ''Argon2'' are deliberately expensive to compute, making brute-force attacks impractical. Argon2 additionally requires large amounts of memory, defending against GPU-based attacks.
