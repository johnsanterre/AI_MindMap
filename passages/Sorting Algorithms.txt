! Sorting Algorithms

<img src="assets/sorting_algorithms.svg" alt="Diagram comparing sorting algorithms with unsorted-to-sorted array transformation and time complexity table for Bubble Sort, Merge Sort, Quick Sort, Heap Sort, and Radix Sort" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you have a messy bookshelf. You could scan the entire shelf for the smallest book and move it to position one, then find the next smallest and place it in position two, and so on — that is ''selection sort''. Or you could split the shelf in half, sort each half separately, then merge the two sorted halves back together — that is ''merge sort''. Sorting algorithms are recipes for rearranging a collection of items into a defined order, and they sit at the heart of computer science because //nearly every computational task// — searching, database indexing, deduplication, ranking ML model predictions — depends on sorted data. The surprise is not that sorting can be done, but that there is a provable speed limit: no comparison-based sort can do better than $O(n \log n)$ in the worst case.

!! The Core Insight

Every comparison-based sorting algorithm builds a ''decision tree'' — a binary tree where each internal node represents a comparison between two elements and each leaf represents a possible output ordering. With $n$ items there are $n!$ possible orderings (by [[Combinatorics]]), so the decision tree must have at least $n!$ leaves. A binary tree with $L$ leaves has height at least $\log_2 L$, giving the lower bound $\log_2(n!) = \Theta(n \log n)$ by Stirling's approximation. This means merge sort, heap sort, and properly implemented quicksort are //asymptotically optimal// — you cannot beat $O(n \log n)$ comparisons on arbitrary input. The only way around this barrier is to exploit structure beyond pairwise comparisons, as ''radix sort'' and ''counting sort'' do by examining individual digits or keys directly.

!! The Algorithm Zoo

''Bubble sort'' repeatedly walks through the list, swapping adjacent elements that are out of order. It is $O(n^2)$ on average and in the worst case, but $O(n)$ on already-sorted input if an early-termination flag is used. Its simplicity makes it useful for teaching, but it is almost never used in practice.

''Insertion sort'' builds the sorted output one element at a time by inserting each new element into its correct position among the already-sorted prefix. It runs in $O(n^2)$ worst case but $O(n)$ on nearly-sorted data, making it the algorithm of choice for small subarrays (typically $n \leq 16$) inside hybrid sorts like Timsort.

''Merge sort'' is the canonical [[Divide and Conquer]] sorting algorithm. It splits the array in half, recursively sorts each half, and merges the two sorted halves in $O(n)$ time. Total: $T(n) = 2T(n/2) + O(n) = O(n \log n)$. It is ''stable'' (equal elements preserve their original order) and guarantees $O(n \log n)$ in all cases, but requires $O(n)$ extra space.

''Quicksort'' picks a ''pivot'' element, partitions the array into elements less than and greater than the pivot, and recurses on each partition. Average case is $O(n \log n)$, but worst case is $O(n^2)$ when pivots are chosen poorly (e.g., always picking the minimum). Randomized pivot selection or median-of-three strategies make the worst case astronomically unlikely. Quicksort is often faster than merge sort in practice due to better cache locality and lower constant factors.

''Heap sort'' builds a [[Heaps and Priority Queues|max-heap]] from the array in $O(n)$ time, then repeatedly extracts the maximum element. It guarantees $O(n \log n)$ worst-case time with $O(1)$ extra space, but its poor cache behavior makes it slower than quicksort in practice.

''Radix sort'' and ''counting sort'' bypass the comparison lower bound entirely. Counting sort handles integers in range $[0, k)$ in $O(n + k)$ time. Radix sort applies counting sort digit-by-digit across $d$ digits, running in $O(d(n + k))$ — linear in $n$ when $d$ and $k$ are constants.

!! What the Math Is Actually Doing

''The comparison lower bound proof:''

$$\text{Height of decision tree} \geq \log_2(n!) \geq \log_2\left(\frac{n}{e}\right)^n = n\log_2 n - n\log_2 e = \Theta(n \log n)$$

This uses Stirling's approximation $n! \approx \sqrt{2\pi n}\left(\frac{n}{e}\right)^n$.

''Merge sort recurrence:'' The time satisfies $T(n) = 2T(n/2) + cn$. Expanding the recursion tree gives $\log_2 n$ levels, each doing $cn$ total work:

$$T(n) = cn \cdot \log_2 n + O(n) = O(n \log n)$$

''Quicksort expected time:'' Let $C(n)$ be the expected number of comparisons. After choosing a uniformly random pivot, each element pair $(i,j)$ is compared with probability $\frac{2}{|j - i| + 1}$ (they are compared only if one of them is chosen as pivot before any element between them). Summing over all pairs:

$$E[C(n)] = \sum_{i=1}^{n} \sum_{j=i+1}^{n} \frac{2}{j - i + 1} = 2n\sum_{k=2}^{n}\frac{1}{k} \approx 2n \ln n \approx 1.39\, n \log_2 n$$

So randomized quicksort uses about 39% more comparisons than the information-theoretic minimum, but fewer data movements than merge sort.

''Counting sort invariant:'' After processing the count array $C$, $C[v]$ holds the number of elements $\leq v$. Placing element with value $v$ at position $C[v] - 1$ and decrementing $C[v]$ produces a stable sort — equal elements appear in their original order.

!! Sorting in Practice and ML

''Standard library sorts'' almost universally use hybrid algorithms. Python's ''Timsort'' combines merge sort and insertion sort, exploiting natural runs (pre-sorted subsequences) in real-world data. C++ STL uses ''Introsort'' (quicksort + heap sort fallback + insertion sort for small arrays). These hybrids achieve $O(n \log n)$ worst case while being fast on typical inputs.

''ML relevance:'' Sorting appears throughout machine learning. //k-nearest neighbors// requires sorting distances. //Ranking losses// (pairwise, listwise) depend on sort order. //Non-maximum suppression// in object detection sorts bounding boxes by confidence. //Top-k sampling// in language models selects the $k$ highest-probability tokens — effectively a partial sort. //Argsort// operations are ubiquitous in NumPy and PyTorch for indexing.

''Parallel sorting:'' Bitonic sort and odd-even merge sort are designed for parallel architectures. Sorting networks — fixed sequences of comparators — can sort in $O(\log^2 n)$ depth with $O(n \log^2 n)$ comparators, enabling GPU-friendly implementations.

!! When Sorting Algorithms Break Down

''Adversarial inputs for quicksort.'' If the pivot selection strategy is deterministic and known to an adversary, inputs can be crafted to trigger $O(n^2)$ behavior every time. This is not merely theoretical — early versions of C's qsort used the first element as pivot, and sorted or reverse-sorted inputs (common in practice) caused worst-case performance. The fix is randomized pivot selection or median-of-three, which makes worst-case inputs exponentially unlikely but does not eliminate the possibility. For guaranteed $O(n \log n)$, use merge sort or heap sort instead.

''Memory pressure with large datasets.'' Merge sort requires $O(n)$ auxiliary space, which becomes prohibitive when sorting billions of records. External sorting algorithms (multiway merge sort reading from disk) address this but introduce I/O bottleneck costs. Quicksort's in-place partitioning uses only $O(\log n)$ stack space on average, making it the better choice when memory is scarce — but its worst-case stack depth is $O(n)$, which can cause stack overflow without tail-call optimization or explicit depth limiting (as Introsort does by switching to heap sort).

''Stability requirements ignored.'' Many applications require ''stable'' sorting — equal elements must retain their original relative order. Quicksort and heap sort are //not stable// by default, so using them for multi-key sorting (e.g., sort by last name, then by first name) silently corrupts the secondary sort order. Merge sort and Timsort are stable. When stability matters and an unstable algorithm is used, the workaround is to append the original index as a tiebreaker, increasing memory usage by $O(n)$.

!! Technical Details

''In-place merge sort'' variants exist (block merge sort) but have significantly higher constant factors and are rarely used in practice.

''Comparison-optimal sorting:'' The exact minimum number of comparisons $S(n)$ needed to sort $n$ elements is known only for small $n$. For example, $S(5) = 7$, requiring the Ford-Johnson (merge-insertion) algorithm. For large $n$, the gap between $\lceil \log_2(n!) \rceil$ and practical algorithms is small.

''Adaptive sorting:'' Algorithms whose running time adapts to existing order in the input. Timsort exploits natural runs; ''Smoothsort'' (by Dijkstra) achieves $O(n)$ on sorted input and $O(n \log n)$ worst case using a Leonardo heap. The measure of disorder (inversions, runs, oscillations) determines which adaptive algorithm is best.

''Sorting networks:'' The AKS network achieves the optimal $O(\log n)$ depth with $O(n \log n)$ comparators, but constants are so large it is impractical. Batcher's bitonic sort ($O(\log^2 n)$ depth, $O(n \log^2 n)$ comparators) is preferred on GPUs.

''Non-comparison sorts:'' Radix sort runs in $O(w \cdot n)$ where $w$ is the key width in digits. For 32-bit integers, this is effectively $O(4n)$, faster than comparison sorts for large $n$. But it requires keys to be decomposable into digits, limiting applicability to numeric or fixed-length string data.
