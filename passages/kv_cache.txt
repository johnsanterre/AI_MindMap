! KV Cache

<img src="assets/kv_cache.svg" alt="Diagram showing KV cache mechanism: prefill phase stores key-value matrices, decode phase attends new token query against cached keys and values" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you are writing a book one word at a time. Every time you need to choose the next word, you do not go back and re-read the entire manuscript from the beginning. You remember what you have already read &mdash; the characters, the plot threads, the tone &mdash; and you use that memory to write the next word. The ''KV cache'' does exactly the same thing for transformer inference. During autoregressive generation, the model produces one token at a time, and each new token must attend to every previous token. Without any caching, the model would recompute the key and value vectors for all previous tokens at every single step. The KV cache stores those previously computed ''key'' and ''value'' vectors so that each new token only needs to compute its own representations and attend against what is already stored. It is a simple idea that transforms a computationally brutal process into something tractable.

!! The Core Insight

During autoregressive generation, each new token must compute <<link "attention" "Attention">><</link>> over all previous tokens. Without a cache, this means recomputing the key ($K$) and value ($V$) projections for every prior token at every generation step. For a sequence of length $L$ with model dimension $d$, that is $O(L^2 d)$ work per token, and $O(L^3 d)$ total to generate a full sequence.

With a KV cache, you compute the key and value vectors for each token //once// and store them. At each new step, you only compute $Q$, $K$, and $V$ for the single new token, attend against the cached keys and values, and append the new $K$ and $V$ to the cache. The per-token cost drops to $O(Ld)$, and total generation cost drops to $O(L^2 d)$. This is the difference between inference being feasible and inference being impossibly slow.

!! How It Works

The process has two distinct phases:

''Prefill phase:'' The entire input prompt is processed in parallel, just like during training. The model computes $Q$, $K$, and $V$ for all prompt tokens at once. The resulting $K$ and $V$ matrices are stored in the cache. This phase is compute-bound and benefits from GPU parallelism.

''Decode phase:'' New tokens are generated one at a time. For each new token, the model computes $Q_{\text{new}}$, $K_{\text{new}}$, and $V_{\text{new}}$ &mdash; each just a single vector. The new token's query attends against the full cached $K$ and $V$ matrices. Then $K_{\text{new}}$ and $V_{\text{new}}$ are appended to the cache. This phase is memory-bandwidth-bound because it reads the entire cache for each token but does relatively little arithmetic.

The cache grows linearly with sequence length. Its total size is:

$$\text{Cache size} = 2 \times n_{\text{layers}} \times n_{\text{heads}} \times d_{\text{head}} \times L \times \text{bytes per element}$$

The factor of 2 accounts for storing both keys and values. For large models at long contexts, this memory cost dominates everything else.

!! What the Math Is Actually Doing

Standard multi-head <<link "attention" "Attention">><</link>> computes, for each head:

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{Q K^T}{\sqrt{d_k}}\right) V$$

where $Q, K, V \in \mathbb{R}^{L \times d_k}$. During full-sequence processing, all $L$ queries attend to all $L$ keys.

With a KV cache during the decode phase, only the new token's query is computed:

$$Q_{\text{new}} \in \mathbb{R}^{1 \times d_k}, \quad K_{\text{cache}} \in \mathbb{R}^{L \times d_k}, \quad V_{\text{cache}} \in \mathbb{R}^{L \times d_k}$$

$$\text{Output} = \text{softmax}\!\left(\frac{Q_{\text{new}} \, K_{\text{cache}}^T}{\sqrt{d_k}}\right) V_{\text{cache}}$$

This produces a single output vector &mdash; the attention output for the new token &mdash; in $O(Ld_k)$ instead of $O(L^2 d_k)$.

To grasp why the KV cache is //the// bottleneck in modern inference, consider a concrete example. A 70B parameter model might have 80 layers, 64 attention heads, and $d_{\text{head}} = 128$. At 128K context length in fp16 (2 bytes per element):

$$\text{Cache} = 2 \times 80 \times 64 \times 128 \times 131072 \times 2 \approx 2.68 \text{ TB}$$

That is more memory than the model weights themselves. The KV cache, not the parameters, is what makes long-context inference so expensive.

!! KV Cache Optimizations

Several techniques have been developed to tame the KV cache memory problem:

''Multi-Query Attention (MQA):'' Instead of separate $K$ and $V$ projections per head, all heads share a single set of keys and values. Only $Q$ differs across heads. This shrinks the cache by a factor of $n_{\text{heads}}$ &mdash; for a 64-head model, that is a 64x reduction. The quality loss is surprisingly small.

''Grouped Query Attention (GQA):'' A middle ground between full multi-head attention and MQA. Heads are divided into groups, and each group shares one set of $K$, $V$ projections. With 8 groups across 64 heads, the cache shrinks by 8x while retaining more representational capacity than MQA. <<link "LLaMA" "Large Language Models">><</link>> 2 70B and many subsequent models use GQA.

''PagedAttention (vLLM):'' Manages KV cache memory the way an operating system manages virtual memory. Instead of pre-allocating contiguous memory for each sequence's full potential length, cache blocks are allocated in fixed-size pages on demand. This eliminates memory fragmentation and allows near-perfect memory utilization across variable-length sequences in a batch.

''Quantized KV cache:'' Store cached keys and values in int8 or int4 instead of fp16, reducing memory by 2&ndash;4x. Because the cached values are only used for attention computation (not gradient updates), aggressive <<link "quantization" "Quantization">><</link>> is tolerable.

''Sliding window attention:'' Only cache the most recent $W$ tokens instead of the full history. Tokens beyond the window are discarded. This caps cache memory at a fixed size regardless of total sequence length, at the cost of losing direct attention to distant context. Models like Mistral combine sliding window layers with full-attention layers.

!! When KV Cache Breaks Down

''Memory wall at long contexts.'' As sequence length grows, the KV cache dominates GPU memory. At 128K context for a 70B model, the raw cache requirement exceeds the memory needed for the model weights themselves. Even with GQA and quantization, serving long-context requests requires either model parallelism across many GPUs or aggressive approximation techniques. The cache is what makes the jump from 4K to 128K context so much harder than it looks.

''Batch size versus cache memory.'' In production serving, you want large batch sizes to amortize the cost of loading model weights from memory. But each sequence in the batch needs its own KV cache. Double the batch size, double the cache memory. At some point you run out of GPU memory, forcing smaller batches and lower throughput. This creates a fundamental tension between latency (larger batches) and the ability to serve long contexts.

''Cache invalidation complexity.'' Advanced inference techniques such as <<link "speculative decoding" "Speculative Decoding">><</link>> and tree-based search generate multiple candidate continuations from the same prefix. Each branch needs its own cache state, which means managing cache trees rather than simple linear caches. Rolling back, forking, and garbage-collecting cache branches adds significant engineering complexity and can negate some of the memory savings from careful cache management.

!! Technical Details

The KV cache concept emerged naturally from the original <<link "Transformer" "Transformers">><</link>> architecture (Vaswani et al., 2017), but its importance only became central with the rise of large autoregressive language models. Key papers addressing KV cache efficiency:

&bull; ''Shazeer 2019'' &mdash; introduced Multi-Query Attention, the first major KV cache compression technique.
&bull; ''Ainslie et al. 2023 (GQA)'' &mdash; proposed Grouped Query Attention as a quality-preserving middle ground between MQA and standard multi-head attention.
&bull; ''Kwon et al. 2023 (vLLM)'' &mdash; introduced PagedAttention, transforming KV cache memory management and enabling dramatically better serving throughput.
