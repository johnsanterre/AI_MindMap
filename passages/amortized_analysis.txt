! Amortized Analysis

<img src="assets/amortized_analysis.svg" alt="Dynamic array insertion costs showing rare expensive resizes averaged to O(1)" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Imagine you have a piggy bank that you drop a coin into every day. Most days, it costs you nothing extra — just one coin in. But every so often the piggy bank fills up, and you have to buy a bigger one, move all the coins over, and then drop your new coin in. That move is expensive — but it happens so rarely that if you spread that cost across all the days, each day is still basically just one coin. That is ''amortized analysis'': instead of panicking about the occasional expensive operation, you average the total cost over the entire sequence and discover the //true// per-operation cost. A dynamic array that doubles its capacity on overflow has a worst-case $O(n)$ insertion, but the amortized cost per insertion is only $O(1)$.

!! The Core Insight

Worst-case per-operation analysis can be deeply misleading when expensive operations are rare and are "paid for" by the many cheap operations surrounding them. Amortized analysis asks: what is the average cost per operation over a ''worst-case sequence'' of $n$ operations?

$$\hat{c} = \frac{\text{Total cost of } n \text{ operations}}{n}$$

This is //not// average-case analysis (which assumes a probability distribution over inputs). Amortized analysis is a worst-case guarantee — it holds for every possible sequence of operations. There are three standard methods for computing amortized costs: the ''aggregate method'', the ''accounting method'', and the ''potential method''.

!! The Three Methods

''1. Aggregate Method'' — the simplest. Compute the total cost $T(n)$ of $n$ operations in the worst case, then divide by $n$. For a dynamic array that doubles: the total cost of $n$ insertions is at most $n + 1 + 2 + 4 + \cdots + 2^{\lfloor \log n \rfloor} < n + 2n = 3n$. So the amortized cost is $3n / n = 3 = O(1)$.

''2. Accounting Method'' — assign each operation an //amortized charge//. Cheap operations are overcharged; the surplus is banked as "credit" on the data structure. When an expensive operation occurs, it withdraws from that credit. The constraint: credits must never go negative. For dynamic arrays, charge each insertion 3 units — 1 for the insertion itself, and 2 banked. When doubling occurs, the banked credits pay for copying.

''3. Potential Method'' — the most powerful and general. Define a ''potential function'' $\Phi$ that maps each state of the data structure to a non-negative real number. The amortized cost of operation $i$ is:

$$\hat{c}_i = c_i + \Phi(D_i) - \Phi(D_{i-1})$$

where $c_i$ is the actual cost and $D_i$ is the state after operation $i$. The potential captures "stored work" — energy built up by cheap operations that will pay for future expensive ones.

!! What the Math Is Actually Doing

The potential method is an elegant bookkeeping trick. Summing amortized costs over $n$ operations:

$$\sum_{i=1}^{n} \hat{c}_i = \sum_{i=1}^{n} c_i + \Phi(D_n) - \Phi(D_0)$$

If $\Phi(D_n) \geq \Phi(D_0)$ (which we ensure by design), then the total amortized cost is an //upper bound// on the total actual cost:

$$\sum_{i=1}^{n} c_i \leq \sum_{i=1}^{n} \hat{c}_i$$

''Dynamic array example:'' Let $\Phi = 2n - \text{capacity}$, where $n$ is the number of elements stored. After a doubling (capacity goes from $m$ to $2m$ and we had $m$ elements), the potential drops from $2m - m = m$ to $2m - 2m = 0$. That drop of $m$ pays for copying $m$ elements. Each normal insertion increases $\Phi$ by 2, which stores work for the next doubling.

''Hash table rehashing:'' With load factor $\alpha$, rehash when $\alpha > 1$. The same doubling argument gives $O(1)$ amortized insertion. This is why hash maps in practice (Python dicts, C++ unordered_map) are fast despite occasional rehashes.

''Fibonacci heap decrease-key:'' Uses a potential function $\Phi = t + 2m$ where $t$ is the number of trees and $m$ is the number of marked nodes. This yields $O(1)$ amortized decrease-key, which is critical for the performance of <<link "Dijkstra's algorithm" "Dijkstras Algorithm">><</link>> and <<link "Prim's algorithm" "Prims Algorithm">><</link>>.

!! Amortized Analysis in ML

Amortized thinking appears throughout machine learning systems, often implicitly:

''Replay buffer sampling:'' In <<link "deep reinforcement learning" "Deep Reinforcement Learning">><</link>>, experience replay buffers store transitions and sample mini-batches. The buffer is a dynamic data structure — insertions are $O(1)$, but periodic reshuffling or priority updates (as in prioritized experience replay) have higher cost. Amortized over many training steps, the per-step overhead is constant.

''Dynamic batching:'' Serving systems like TensorFlow Serving accumulate requests and batch them for GPU inference. Most requests wait cheaply; occasionally a batch is dispatched with higher latency. The amortized throughput per request is much better than processing each individually.

''Lazy evaluation in computation graphs:'' Frameworks like <<link "JAX" "Jax">><</link>> and TensorFlow defer computation until results are needed, then execute an optimized fused kernel. Individual operation "costs" are near-zero (just graph construction), with the real work amortized across the fused execution.

''Gradient accumulation:'' When GPU memory is too small for a large batch, gradients are accumulated over multiple forward passes before a single weight update. Each forward pass is cheap; the optimizer step is expensive but amortized over many micro-batches.

''Hash-based approximate nearest neighbors:'' Methods like locality-sensitive hashing (LSH) amortize the cost of building hash tables across many queries. Building is $O(n)$, but each query is $O(1)$ amortized, making it practical for nearest-neighbor search in <<link "embedding" "Embeddings">><</link>> spaces.

''Incremental PCA updates:'' Rather than recomputing <<link "PCA" "Principal Component Analysis">><</link>> from scratch when new data arrives, incremental methods amortize the update cost, processing each new sample in $O(k)$ where $k$ is the number of components.

!! When Amortized Analysis Breaks Down

''1. Real-time systems need per-operation guarantees.'' If you are running inference in a self-driving car, an occasional $O(n)$ operation is unacceptable even if the amortized cost is $O(1)$. Real-time systems require //worst-case per-operation// bounds, not amortized ones. Solutions include de-amortized data structures that spread expensive work incrementally across cheap operations.

''2. Adversarial inputs can force worst-case every time.'' Amortized analysis assumes worst-case //sequences//, but an adversary who knows the internal state can trigger expensive operations back-to-back. For example, repeatedly inserting and deleting around the resize threshold of a dynamic array can cause it to resize on every operation, destroying the amortized guarantee. Randomized strategies (like randomized doubling thresholds) can defend against this.

''3. Amortized bounds do not compose freely.'' If data structure A has $O(1)$ amortized insert and data structure B has $O(1)$ amortized insert, interleaving operations on both does //not// guarantee $O(1)$ amortized overall. The potential functions can interfere. When one structure's expensive operation triggers during another's cheap phase, the combined cost can be $O(n)$. Composing amortized bounds requires analyzing the combined potential function.

!! Technical Details

''Origin:'' The term "amortized analysis" was formalized by Robert Tarjan in 1985, building on earlier work by Sleator, Tarjan, and others on self-adjusting data structures. The potential method was directly inspired by physics — potential energy stored in a system.

''Splay trees:'' Tarjan and Sleator's splay trees achieve $O(\log n)$ amortized cost for search, insert, and delete, despite individual operations potentially taking $O(n)$. The potential function $\Phi = \sum \log s(x)$ where $s(x)$ is the size of the subtree rooted at $x$ provides the bound. See <<link "balanced BSTs" "Balanced Binary Search Trees">><</link>>.

''Union-Find:'' The disjoint-set data structure with union by rank and path compression achieves $O(\alpha(n))$ amortized per operation, where $\alpha$ is the inverse Ackermann function — a function that grows so slowly it is effectively constant for all practical inputs. This is proved using a carefully designed potential function.

''Multipop stacks:'' A stack that supports push ($O(1)$), pop ($O(1)$), and multipop($k$) (pop $k$ elements, $O(k)$ worst case) has $O(1)$ amortized cost per operation. The accounting method charges 2 for each push: 1 for pushing, 1 banked for a future pop.

''CLRS textbook:'' Chapter 17 of //Introduction to Algorithms// (Cormen, Leiserson, Rivest, Stein) provides the standard pedagogical treatment. See also: <<link "Big O notation" "Big O Notation">><</link>>, <<link "Big O examples" "Big O Examples">><</link>>, <<link "complexity classes" "Complexity Classes">><</link>>, and <<link "algorithm design paradigms" "Algorithm Design Paradigms">><</link>>.
