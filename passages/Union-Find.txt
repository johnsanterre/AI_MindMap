! Union-Find

<img src="assets/union_find.svg" alt="Diagram showing Union-Find disjoint set forest before and after union with path compression">

Imagine you are at a large party where people keep discovering mutual friends. Two strangers shake hands, and suddenly their entire friend groups merge. Later someone asks, "Do Alice and Zara know each other through anyone here?" You do not need to trace every handshake — you just need to know whether they ended up in the same group. ''Union-Find'' (also called the ''disjoint-set'' data structure) solves exactly this problem: it maintains a collection of non-overlapping sets and supports two operations — merging two sets together and querying whether two elements belong to the same set — both in nearly constant time.

!! The Core Insight

Union-Find represents each set as a rooted [[Trees and Forests|tree]]. Every element points to a parent, and the ''root'' of the tree serves as the set's representative. Two elements are in the same set if and only if they share the same root. The genius is in two simple optimizations — ''union by rank'' and ''path compression'' — that keep the trees almost completely flat. Union by rank always attaches the shorter tree under the taller one. Path compression rewires every node along a Find path to point directly at the root. Together, these reduce the amortized cost per operation to $O(\alpha(n))$, where $\alpha$ is the inverse Ackermann function — a function that grows so extraordinarily slowly that it never exceeds 4 for any input size that could physically exist.

!! How the Forest Grows

'''Initialization.''' Every element starts as its own singleton tree: it is its own parent, and its rank is zero. With $n$ elements, you begin with $n$ disjoint sets.

'''Find(x).''' Walk from $x$ up the parent pointers until you reach a node that is its own parent — that is the root. Along the way, path compression rewires every visited node to point directly to the root, so future queries on those nodes are nearly instant.

'''Union(x, y).''' First, find the roots of $x$ and $y$. If they are already the same root, done — $x$ and $y$ are in the same set. Otherwise, attach the root of smaller rank under the root of larger rank. If ranks are equal, pick either one as the new root and increment its rank by one.

'''Why it works for graphs.''' When building a [[Minimum Spanning Trees|minimum spanning tree]] with Kruskal's algorithm, you process edges in order of weight and add each edge only if its endpoints are in different components. Union-Find answers "different components?" with Find and merges them with Union — making Kruskal's algorithm run in $O(E \log E)$ time, dominated by sorting the edges rather than the set operations.

!! What the Math Is Actually Doing

Each element $x$ maintains a parent pointer $\text{parent}[x]$ and an integer $\text{rank}[x]$. The operations are defined as:

$$\text{Find}(x) = \begin{cases} x & \text{if } \text{parent}[x] = x \\ \text{Find}(\text{parent}[x]) & \text{otherwise} \end{cases}$$

with the side effect that $\text{parent}[x] \leftarrow \text{Find}(\text{parent}[x])$ (path compression).

For Union, let $r_x = \text{Find}(x)$ and $r_y = \text{Find}(y)$:

$$\text{Union}(x,y): \quad \text{parent}[r_x] \leftarrow r_y \;\;\text{if}\;\; \text{rank}[r_x] < \text{rank}[r_y], \quad \text{else swap}$$

The ''amortized'' complexity proof uses a potential function based on Ackermann's function. Define $A_k(j)$ as a rapidly growing hierarchy of functions:

$$A_0(j) = j + 1, \qquad A_k(j) = A_{k-1}^{(j+1)}(j) \;\;\text{for } k \ge 1$$

where the superscript denotes iterated application. The inverse $\alpha(n) = \min\{k : A_k(1) \ge n\}$ grows so slowly that $\alpha(2^{65536}) = 4$. A carefully designed potential function over node ranks and levels proves that $m$ operations on $n$ elements cost $O(m \cdot \alpha(n))$ total.

!! Union-Find in Practice

'''Kruskal's MST.''' Sort edges by weight, iterate through them, and use Union-Find to check whether each edge connects two different components. If so, add it to the spanning tree and union the components. This is the textbook application and directly leverages the efficiency of the data structure. See [[Minimum Spanning Trees]].

'''Connected components.''' For a static undirected graph, Union-Find computes all connected components in $O((V + E)\,\alpha(V))$ time by iterating over edges and unioning endpoints. This rivals BFS/DFS and is simpler to implement in parallel or distributed settings.

'''Image segmentation.''' Region-merging algorithms in computer vision start with each pixel as its own component and merge adjacent pixels whose color difference falls below a threshold. The resulting components are the segments. Felzenszwalb's efficient graph-based segmentation algorithm uses Union-Find at its core.

'''Percolation theory.''' In physics simulations, Union-Find determines whether a grid "percolates" — whether there is a connected path from top to bottom. Opening sites one at a time and unioning with open neighbors gives an $O(n\,\alpha(n))$ algorithm for the percolation threshold.

'''Equivalence classes in compilers.''' Type inference and register allocation use Union-Find to track which variables must have the same type or share the same storage. Hindley-Milner type inference in functional languages relies on unification, which is essentially Union-Find on type expressions.

!! When Union-Find Breaks Down

* ''Need to undo merges.'' Standard Union-Find is a //one-way// operation — once two sets are merged, there is no efficient way to split them apart. If your algorithm requires dynamic splitting (e.g., maintaining components under edge deletions), Union-Find cannot help without a full rebuild. Offline solutions exist using "link-cut trees" or the retrospective approach of processing deletions in reverse as insertions.

* ''Need to enumerate set members.'' Union-Find tells you //whether// two elements share a set and //who the representative is//, but it does not efficiently list all members of a set. If you need to iterate over every element in a component, you must maintain an auxiliary data structure (e.g., a linked list threaded through each set), which adds $O(n)$ overhead per merge.

* ''Concurrent access without care.'' Naive parallelization of Union-Find introduces race conditions: two threads may simultaneously try to update the same parent pointer. Lock-free and wait-free variants exist (using compare-and-swap), but they sacrifice path compression or require weaker amortized guarantees. Careless parallelism can silently corrupt the forest.

!! Technical Details

''Space complexity:'' $O(n)$ — two arrays of length $n$ (parent and rank).

''Without optimizations:'' Union-Find with neither union by rank nor path compression can degenerate to $O(n)$ per Find (a single long chain). Union by rank alone gives $O(\log n)$ per Find. Path compression alone gives $O(\log n)$ amortized. Both together achieve $O(\alpha(n))$.

''Weighted Union-Find:'' Each edge can carry a weight representing the relative difference between elements. This variant tracks $\text{weight}[x]$, the offset from $x$ to its parent, and computes the relative offset between any two elements in the same set during Find. Used in problems like "element $a$ is $k$ units greater than element $b$."

''Persistent Union-Find:'' Using a persistent array (e.g., a balanced BST of modifications), Union-Find can be made fully persistent in $O(\log n)$ per operation, allowing queries against any historical version of the data structure.

''Relation to [[Graph Theory]]:'' Union-Find is fundamentally a dynamic connectivity structure for //incremental// (insertion-only) undirected graphs. It cannot handle decremental connectivity efficiently in its standard form.

''Fredman and Saks lower bound (1989):'' Any data structure for the disjoint-set problem requires $\Omega(m \cdot \alpha(n))$ time for $m$ operations in the cell-probe model, proving that Union-Find with both optimizations is //optimal//.
