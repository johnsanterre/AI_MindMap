! Retrieval Augmented Generation

<img src="assets/retrieval_augmented_generation.svg" alt="Retrieval Augmented Generation diagram showing query encoding, vector search, context augmentation, and grounded LLM output" style="max-width:100%; height:auto;">

Imagine you are taking an open-book exam. You know how to read, reason, and write coherent answers, but instead of relying purely on what you memorized the night before, you flip through your textbook and notes to find the relevant passages first. That is exactly what ''Retrieval Augmented Generation'' (RAG) does for a large language model: it gives the model an open book. The model's parametric memory — the billions of weights frozen at training time — is supplemented at inference time with a live retrieval step that fetches relevant documents from an external knowledge store. The result is an answer grounded in actual source material rather than a confident-sounding guess.

!! The Core Insight

A language model's knowledge is ''static and implicit'' — baked into its parameters at training time, impossible to audit, and expensive to update. RAG decouples ''what the model knows how to do'' (reason, summarize, compose) from ''what it knows about'' (facts, figures, current events). By retrieving relevant context on the fly and injecting it into the prompt, you get three things at once: reduced <<link "Hallucination" "Hallucination">><</link>>, easy knowledge updates (just refresh the document store), and the ability to cite sources — something a vanilla language model cannot do.

The architectural insight is that retrieval and generation are complementary, not competing. Retrieval excels at precise factual recall over huge corpora; generation excels at synthesis, reasoning, and fluent composition. RAG combines both strengths in a single pipeline.

!! How the Pipeline Works

A RAG system has two main phases: ''indexing'' (offline) and ''query-time retrieval + generation'' (online).

'''Indexing phase:'''
Documents are chunked into passages (typically 100–500 tokens), each passage is run through an embedding model to produce a dense vector representation (see <<link "Embeddings" "Embeddings">><</link>>), and the vectors are stored in a vector database alongside the raw text.

'''Query-time phase:'''
# The user's query is embedded using the same encoder.
# The query vector is compared against all stored document vectors using <<link "Cosine Similarity" "Cosine Similarity">><</link>> (or another distance metric) to find the top-$k$ most relevant chunks.
# Those chunks are prepended to the query as context inside the prompt.
# The <<link "Transformers" "Transformers">><</link>>-based LLM generates an answer conditioned on both the query and the retrieved context.

The key design choice is how many chunks to retrieve ($k$), how to chunk documents, and which embedding model to use. These choices profoundly affect answer quality.

!! What the Math Is Actually Doing

The retrieval step is a ''nearest-neighbor search in embedding space''. Given a query $q$ and a corpus of $N$ document chunks $\{d_1, d_2, \ldots, d_N\}$, we first encode them:

$$\mathbf{q} = \text{Encode}(q), \quad \mathbf{d}_i = \text{Encode}(d_i)$$

Relevance is scored by cosine similarity:

$$\text{score}(q, d_i) = \frac{\mathbf{q} \cdot \mathbf{d}_i}{\|\mathbf{q}\| \, \|\mathbf{d}_i\|}$$

We select the top-$k$ documents:

$$\mathcal{R} = \text{top-}k \left( \{ \text{score}(q, d_i) \}_{i=1}^{N} \right)$$

The generation step conditions on retrieved context. If $\theta$ denotes model parameters and $x$ the query, the output distribution becomes:

$$P(y \mid x) = \sum_{d \in \mathcal{R}} P(d \mid x) \cdot P_\theta(y \mid x, d)$$

In practice, most systems simply concatenate the retrieved chunks into the prompt rather than marginalizing, giving:

$$P(y \mid x) \approx P_\theta(y \mid [d_1; d_2; \ldots; d_k; x])$$

This concatenation approach is simpler and works well when the retriever is accurate. The more principled marginalization approach, used in the original RAG paper by Lewis et al. (2020), treats the retrieved document as a latent variable and trains retriever and generator end-to-end.

!! Chunking, Indexing, and Reranking

The practical success of RAG depends heavily on engineering details:

''Chunking strategy:'' Documents must be split into passages small enough to be semantically coherent but large enough to carry useful context. Common strategies include fixed-size token windows with overlap, recursive splitting by headers or paragraphs, and semantic chunking based on embedding similarity breakpoints.

''Hybrid retrieval:'' Dense vector search can miss exact keyword matches. Many production systems combine dense retrieval with sparse retrieval (BM25 / TF-IDF) and merge the ranked lists — a technique called hybrid search.

''Reranking:'' The initial retrieval uses fast approximate search. A cross-encoder reranker can then rescore the top-$k$ candidates with higher accuracy, since cross-encoders attend jointly to query and document but are too expensive to run over the entire corpus.

''Metadata filtering:'' Adding structured metadata (date, source, topic) lets you pre-filter the corpus before vector search, improving relevance and reducing noise.

<<link "Knowledge Graphs" "Knowledge Graphs">><</link>> can also serve as a retrieval source, providing structured relational facts that complement unstructured text retrieval.

!! When X Breaks Down

''1. Retrieval failure (garbage in, garbage out):'' If the retriever returns irrelevant or misleading chunks — due to poor embeddings, bad chunking, or domain mismatch — the generator will faithfully incorporate that noise into its answer. The model cannot easily distinguish retrieved truth from retrieved nonsense. This is the single most common RAG failure mode.

''2. Lost in the middle:'' Research shows that LLMs attend more strongly to information at the beginning and end of long contexts, neglecting material in the middle. When many chunks are retrieved, the most relevant passage may land in a low-attention zone and be effectively ignored, even though it was successfully retrieved.

''3. Context window overflow:'' Every retrieved chunk consumes prompt tokens. Retrieve too many and you exceed the model's context window or crowd out the actual query. Retrieve too few and you risk missing the answer. There is no free lunch — chunk count $k$, chunk size, and context length are in constant tension.

!! Technical Details

''Original RAG paper:'' Lewis et al. (2020) introduced two variants: ''RAG-Sequence'' (one document conditions the entire output) and ''RAG-Token'' (different documents can condition different tokens). Both train the retriever (DPR) and generator (BART) end-to-end.

''Retriever architectures:'' Dual-encoder (bi-encoder) models embed query and document independently for fast search. Cross-encoders process them jointly for higher accuracy but $O(N)$ cost. Dense Passage Retrieval (DPR) is the canonical dual-encoder.

''Approximate nearest neighbor (ANN):'' Exact cosine search is $O(N)$. Production systems use ANN indices (FAISS, ScaNN, HNSW) for sub-linear search with controllable recall loss.

''Advanced variants:''
* ''Self-RAG:'' The model decides when to retrieve and critiques its own outputs.
* ''Corrective RAG (CRAG):'' A lightweight evaluator scores retrieved documents; if scores are low, the system falls back to web search.
* ''RAPTOR:'' Recursively summarizes and clusters chunks into a tree, enabling retrieval at multiple abstraction levels.
* ''Hypothetical Document Embeddings (HyDE):'' The LLM first generates a hypothetical answer, which is then used as the retrieval query — often more effective than using the raw question.

''Evaluation metrics:'' RAG systems are evaluated on both retrieval quality (recall@k, MRR) and generation quality (faithfulness, answer relevance, context precision). Frameworks like RAGAS formalize these into composite scores.

''Embedding model choice:'' The quality of the embedding model is the single largest lever. Models like E5, GTE, and BGE are specifically trained for retrieval and outperform general-purpose <<link "Embeddings" "Embeddings">><</link>> on retrieval benchmarks.
