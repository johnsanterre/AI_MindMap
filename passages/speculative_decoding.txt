! Speculative Decoding

<img src="assets/speculative_decoding.svg" alt="Diagram illustrating Speculative Decoding with a small draft model and large target model" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

Proofreading is faster than writing from scratch. Imagine a junior editor who drafts a paragraph quickly, then a senior editor who reviews the whole thing in one pass — fixing only the sentences they disagree with. Speculative decoding works exactly like this: a small, fast ''draft model'' generates several candidate tokens in sequence, and the large ''target model'' checks all of them in a single parallel forward pass, accepting the ones it agrees with and resampling only the ones it would have chosen differently. The output you receive is statistically identical to what the large model would have produced on its own — just delivered much faster.

---

!! The Core Insight

Large language model inference is ''memory-bandwidth bound'', not compute bound. The bottleneck is not arithmetic — modern GPUs have arithmetic to spare — but the cost of loading billions of model weights from GPU memory for every single autoregressive step. Generating 100 tokens from a 70B-parameter model requires 100 serial forward passes, each loading the full weight matrix. That is slow regardless of how fast your hardware multiplies numbers.

Speculative decoding breaks this serialization. A small draft model — perhaps 7B parameters, ten times cheaper per call — generates K candidate tokens one at a time. The large target model then evaluates all K tokens in a //single// forward pass, because the transformer attention mechanism can score all positions simultaneously given the draft sequence as context. If most drafts are accepted, you receive K tokens for approximately the cost of one target model call. At an acceptance rate of 90% with K=4 draft tokens, you approach a 4x wall-clock speedup.

The key guarantee that makes speculative decoding practical rather than merely fast is its ''exactness'': the joint output distribution over accepted tokens is mathematically identical to sampling from the target model directly. There is no approximation, no degradation in output quality, no trade-off between speed and correctness. This distinguishes speculative decoding from quantization or distillation, which sacrifice fidelity for speed. See [[Large Language Models]], [[Transformers]], [[Quantization]], and [[Hardware Lottery]] for related context.

---

!! The Acceptance-Rejection Scheme

The elegance of speculative decoding lies in its acceptance criterion, which is a variant of classical ''rejection sampling''. For each draft token $x$ at position $t$, let $q(x)$ denote the probability the draft model assigned to that token, and $p(x)$ denote the probability the target model assigns. The token is accepted with probability:

$$\alpha(x) = \min\!\left(1,\, \frac{p(x)}{q(x)}\right)$$

When $p(x) \geq q(x)$ — that is, when the target model likes the draft token at least as much as the draft model does — the token is always accepted. When the draft model was //overconfident// about a token the target model disfavors, the token is accepted with reduced probability proportional to how much the target disagrees.

If a token is rejected, a replacement is not drawn from the full target distribution $p$. Instead it is drawn from the ''corrected distribution'' $p'(x) \propto \max(0,\, p(x) - q(x))$, which removes the probability mass already covered by $q$ and redistributes it. This correction ensures that the marginal distribution over each output token is exactly $p$, regardless of what the draft model did. The entire sequence of operations — draft, verify, accept/reject, correct — produces samples that are provably equivalent to ancestral sampling from the target model.

---

!! What the Math Is Actually Doing

The per-token acceptance probability is:

$$\alpha = \mathbb{E}\!\left[\min\!\left(1,\, \frac{p(x)}{q(x)}\right)\right]$$

where the expectation is over tokens drawn from the draft distribution $q$. This is a number between 0 and 1 that summarizes how well the draft model approximates the target at each step.

Given K draft tokens and a per-token acceptance rate $\alpha$, the expected number of tokens produced per target model forward pass is:

$$\mathbb{E}[\text{tokens per call}] = \frac{1 - \alpha^{K+1}}{1 - \alpha}$$

This counts the accepted tokens plus one guaranteed token: even if all K drafts are rejected, the corrected distribution yields exactly one token from the target. At $\alpha = 0.9$ and $K = 4$, the expected output is approximately $3.8$ tokens per target call — close to the theoretical maximum of 4. At $\alpha = 0.7$ and $K = 4$, the expected output drops to about $2.6$ tokens per call.

A meaningful speedup requires two conditions simultaneously: the draft model must be ''much faster'' than the target (achieved by a large size ratio, e.g., 7B vs 70B), and the draft model must have a ''high acceptance rate'' with the target (achieved by using a model from the same training family). If either condition fails — the draft is too slow, or $\alpha$ is too low — speculative decoding may be slower than standard sampling.

---

!! Variants: Self-Speculation and Medusa

The original formulation assumes a separate, smaller draft model. In practice, several variations trade different resources against different benefits.

''Same-family drafting'' — using LLaMA-7B to draft for LLaMA-70B, for instance — achieves high acceptance rates because both models share training data and objectives. The draft distribution $q$ closely tracks the target distribution $p$ across most common token sequences, yielding $\alpha$ values above 0.8 in typical text generation benchmarks.

''Medusa'' (Cai et al., 2024) eliminates the separate draft model entirely by attaching multiple lightweight prediction heads directly to the large model's final hidden layer. Each head predicts a future token position in parallel, producing K drafts in a single forward pass of the large model at the cost of a small fraction of additional computation. This avoids the memory overhead of maintaining two models simultaneously, at the cost of slightly lower acceptance rates than a well-matched external draft model.

''Eagle'' (Li et al., 2024) goes further by feeding the large model's //hidden states// — not just its output logits — as input to the draft mechanism. Because the hidden states carry rich contextual information about the target model's internal representations, Eagle achieves very high acceptance rates, often exceeding Medusa. ''Layer-skipping'' is another self-speculative approach: the large model itself generates drafts using only a subset of its layers, then verifies with all layers. This keeps everything within a single model at the cost of reduced draft quality compared to layer-skipping.

---

!! When Speculative Decoding Breaks Down

''Low acceptance rate destroys the speedup.'' When the draft model's distribution diverges substantially from the target's — for instance, when the draft model is from a different training lineage, or when the prompt is highly specialized (code, mathematics, low-resource languages) — rejection rates climb. At $\alpha = 0.5$ with K=4, the expected tokens per target call falls to about 1.9, barely better than standard sampling once draft model overhead is included. Speculative decoding offers no free lunch: if the draft model cannot approximate the target, the verification pass is wasted.

''Large batch inference erases the bandwidth bottleneck.'' The memory-bandwidth argument — that each token requires reloading weights — assumes a batch size of one or a small number of concurrent requests. At large batch sizes, the GPU's arithmetic units are already saturated and weight loading is amortized across many sequences in parallel. In high-throughput production serving with hundreds of concurrent requests, the marginal gain from speculative decoding shrinks substantially, and the additional complexity of maintaining draft models may not be worth the engineering cost.

''Simultaneous VRAM residency constrains deployment.'' Running speculative decoding requires both the draft model and the target model to reside in GPU memory at the same time. A 70B target model already pushes the limits of multi-GPU inference setups; adding a 7B draft model imposes a non-trivial additional memory footprint. Self-speculative methods (Medusa, Eagle) avoid this constraint, but the original two-model formulation can be impractical on memory-limited hardware, limiting its applicability to settings where VRAM is abundant.

---

!! Technical Details

The original speculative decoding paper is Leviathan, Kalman, and Matias (2022), "Fast Inference from Transformers via Speculative Decoding." An independent contemporaneous formulation appeared as Chen et al. (2023), "Accelerating Large Language Model Decoding with Speculative Sampling," from Google DeepMind. Both arrive at the same acceptance-rejection scheme from slightly different angles.

Medusa is described in Cai et al. (2024), "Medusa: Simple LLM Inference Acceleration Framework with Multiple Decoding Heads." Eagle is described in Li et al. (2024), "Eagle: Speculative Sampling Requires Rethinking Feature Uncertainty." Eagle-2 (also Li et al., 2024) further improves on Eagle by dynamically adapting the draft length K based on context-dependent acceptance predictions.

The token budget K is a hyperparameter with a non-trivial optimum. Larger K increases the expected tokens per target call when $\alpha$ is high, but wastes draft model compute when $\alpha$ is low. Adaptive K selection — halting the draft early if confidence drops — is an active area of work. Both models must remain loaded in VRAM simultaneously in the two-model setting, which doubles the memory footprint relative to standalone target inference. The correction distribution $p'(x) \propto \max(0, p(x) - q(x))$ requires computing the full vocabulary distribution from the target model, not just the top token; implementations must ensure the target model returns full logits rather than greedy outputs.
