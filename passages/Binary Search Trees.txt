! Binary Search Trees

<img src="assets/binary_search_trees.svg" alt="Binary Search Trees" width="800">

Imagine walking into a library where every book is shelved in a single long row, sorted by title. To find a specific book, you could start at one end and check every spine — but that's painfully slow. Instead, you walk to the middle of the shelf, check whether your title comes before or after, and immediately ignore half the books. Then you repeat on the remaining half. A ''binary search tree'' (BST) is the data structure that makes this halving strategy permanent: each node stores a value and two pointers (left for "smaller," right for "larger"), creating a shape that lets you throw away half the remaining candidates at every step.

!! The Core Insight

The defining property of a BST is the ''BST invariant'': for every node, all values in its left subtree are strictly less than the node's value, and all values in its right subtree are strictly greater. This single rule transforms an unstructured pile of data into a hierarchy where every comparison eliminates an entire branch. Unlike a sorted array, which gives you fast search but slow insertion (you have to shift elements), a BST provides fast search //and// fast insertion — as long as the tree stays roughly balanced. When it doesn't, you get a degenerate chain that's no better than a linked list, which is why [[Balanced Trees]] like AVL trees and red-black trees exist.

!! Navigating and Modifying the Tree

''Search'' follows the invariant: compare your target to the current node, go left if smaller, go right if larger, stop when you find it or hit a null pointer. ''Insertion'' works the same way — you search for where the value //would// be, and when you reach a null spot, you place the new node there. ''Deletion'' is trickier. If the node to remove has no children, just delete it. If it has one child, splice it out by connecting its parent to its child. If it has two children, you find its ''in-order successor'' (the smallest value in the right subtree) or ''in-order predecessor'' (the largest in the left subtree), swap values, and delete the successor/predecessor instead — which is guaranteed to have at most one child.

''In-order traversal'' — visiting the left subtree, then the node, then the right subtree — produces all values in sorted order. This makes a BST a natural choice when you need data that's always sorted: you can iterate through all $n$ values in $O(n)$ time. Pre-order and post-order traversals serve different purposes, such as serializing the tree or evaluating expression trees.

!! What the Math Is Actually Doing

The expected height of a BST built from $n$ randomly inserted keys is:

$$E[h] = O(\log n)$$

More precisely, it's approximately $4.311 \ln n$ for a random BST. This means that search, insertion, and deletion all run in expected $O(\log n)$ time, because each operation follows a root-to-leaf path. The worst case occurs when keys arrive in sorted or reverse-sorted order, producing a degenerate tree of height $n - 1$:

$$T_{\text{worst}}(n) = O(n)$$

The number of distinct BST shapes for $n$ keys is the $n$-th Catalan number:

$$C_n = \frac{1}{n+1} \binom{2n}{n} = \frac{(2n)!}{(n+1)!\, n!}$$

This grows exponentially — there are 42 distinct BSTs for just 5 keys. The relationship between BSTs and [[Sorting Algorithms]] is direct: building a BST from a sequence and then reading it in-order is equivalent to performing a comparison-based sort, inheriting the $\Omega(n \log n)$ lower bound.

!! Practical Applications

BSTs form the backbone of ordered data structures across computing. Language standard libraries implement ordered maps and sets using self-balancing BSTs — C++'s `std::map` uses a red-black tree, Java's `TreeMap` does the same. Database indexes frequently use B-trees (a generalization of BSTs with more than two children per node), and file systems use them for directory lookups.

In machine learning, [[Trees and Forests]] like decision trees and random forests are conceptual descendants of BSTs. A decision tree splits feature space by asking binary questions at each node — "is feature $x_j < t$?" — creating a BST-like structure over thresholds. The key difference is that decision trees partition a //feature space// for prediction, while BSTs partition a //value space// for retrieval. KD-trees, used for nearest-neighbor search in high-dimensional spaces, extend the BST idea by alternating which dimension is compared at each level.

Augmented BSTs attach extra information to each node. An ''order-statistic tree'' stores subtree sizes, enabling $O(\log n)$ queries like "find the $k$-th smallest element." An ''interval tree'' stores intervals and supports efficient overlap queries. These augmentations preserve the BST structure while adding powerful query capabilities.

!! When Binary Search Trees Break Down

* ''Sorted or nearly sorted input.'' Inserting keys in sorted order produces a tree with height $n - 1$, turning every $O(\log n)$ operation into $O(n)$. This is the BST's Achilles' heel. Self-balancing variants like AVL trees and red-black trees ([[Balanced Trees]]) add rotation operations that maintain $O(\log n)$ height guarantees after every insertion and deletion, at the cost of slightly more complex code and constant-factor overhead.

* ''High-frequency concurrent access.'' Standard BSTs are not thread-safe. Concurrent insertions and deletions can corrupt the tree structure. Lock-based solutions create contention bottlenecks, especially near the root, since every operation starts there. Lock-free and concurrent skip lists often outperform BSTs in multi-threaded environments, and hash tables avoid the tree structure entirely for unordered access patterns.

* ''Cache-unfriendly memory layout.'' BST nodes are typically heap-allocated and scattered across memory. Pointer chasing — following left/right references — causes frequent cache misses, especially in large trees. B-trees address this by packing many keys into each node (matching CPU cache line sizes), dramatically reducing the number of pointer dereferences. For purely in-memory workloads on modern hardware, sorted arrays with binary search often outperform BSTs due to superior cache locality.

!! Technical Details

''Balancing strategies'' fall into two categories. ''Strict balancing'' (AVL trees) guarantees that left and right subtree heights differ by at most 1, giving tighter $O(\log n)$ bounds but requiring more rotations on insert/delete. ''Relaxed balancing'' (red-black trees) allows height differences up to a factor of 2, performing fewer rotations at the cost of slightly taller trees (height $\leq 2 \log_2(n+1)$). ''Randomized'' approaches like treaps assign random priorities and maintain a heap property on priorities alongside the BST property on keys, achieving $O(\log n)$ expected performance without deterministic balancing.

''Splay trees'' are self-adjusting BSTs that move each accessed node to the root via rotations (splaying). They have no explicit balance guarantee, but achieve $O(\log n)$ //amortized// time per operation. Their key strength is temporal locality: frequently accessed elements stay near the root, making them excellent for caches and working-set access patterns.

''Space complexity'' is $O(n)$ for $n$ keys, with each node storing the key, a value (optionally), and two child pointers. In practice, pointer overhead is significant: a node holding a 4-byte integer may require 24+ bytes for pointers and alignment. This motivates implicit BST representations (like sorted arrays with binary search) when the data is static.

The ''BST sort'' equivalence means that any lower bound on comparison sorting applies to BST construction. Building a BST from $n$ arbitrary keys requires $\Omega(n \log n)$ comparisons in the worst case. Conversely, if the keys are drawn uniformly at random, the expected number of comparisons for building the tree is $O(n \log n)$, matching optimal sorting.
