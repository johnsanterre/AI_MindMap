! LLM Agents

<img src="assets/llm_agents.svg" alt="Diagram illustrating the ReAct loop for LLM Agents" style="width:100%;max-width:620px;display:block;margin:16px auto 24px auto;">

A calculator is smart but passive — it only answers when you ask. An ''LLM agent'' is different. It has a goal, makes a plan, picks up tools, checks what they return, and adjusts course. The language model is not just generating text to be read; it is the reasoning engine of a system that acts in the world. It decides to run a web search, inspects the result, decides the answer was incomplete, runs a follow-up query, writes and executes code to crunch the numbers, and only then produces a final answer. That loop — reason, act, observe, repeat — is what separates an agent from a chatbot. The key breakthrough was realizing that language models are surprisingly capable of orchestrating this loop if given the right prompting structure and a well-defined set of tools to call.

---

!! The Core Insight

An LLM agent is an LLM augmented with three things: a set of ''tools'' it can invoke (web search, code execution, APIs, memory retrieval), a ''loop'' that feeds each tool's output back into the model's context, and a ''reasoning strategy'' that decomposes a complex goal into a sequence of manageable subtasks. None of those pieces alone is novel. What is new is combining them with a sufficiently capable language model and showing that the result can autonomously solve multi-step tasks that are completely beyond a single forward pass.

The key insight is that language models trained on code and natural language acquire a latent capacity for tool-use reasoning. Given a description of what a tool does — a function name, a docstring, an example — a large model can reliably decide when to call it, how to format the call, and how to interpret the result. This means the "brain" of the agent does not need to be redesigned for each new tool; you add a tool description to the context and the model figures out the rest. This enables tasks that are impossible for a single forward pass: multi-step research across a live web, code debugging with execution feedback, web-based question answering requiring dozens of retrieval steps, and autonomous software engineering. See [[Large Language Models]], [[Chain of Thought]], [[In-Context Learning]], [[Prompt Engineering]], and [[RAG]] for the building blocks this architecture rests on.

The most important structural feature is the ''context window as working memory''. Every thought the model has produced, every tool call it has made, and every observation the tool returned accumulates in the context. The model reads this entire history before generating its next thought. This gives it the ability to track state across many steps — but also means that very long tasks risk exhausting the context window, a central limitation of current agents.

---

!! ReAct: Reason + Act

''ReAct'' (Yao et al. 2023) is the foundational algorithm for tool-using agents. The name is a portmanteau: //Reason// + //Act//. The idea is to interleave reasoning traces and tool-use actions in the same generation stream, in a strict alternating format:

//Thought: I need to find the current population of France.//<br>
//Action: Search[population of France 2024]//<br>
//Observation: 68 million as of 2024.//<br>
//Thought: Now I can multiply by two.//<br>
//Action: Calculator[68000000 × 2]//<br>
//Observation: 136000000//<br>
//Answer: 136 million.//

The reasoning trace serves two functions. First, it keeps the model on-task by making the plan explicit — the model can re-read its own prior thoughts and see what step comes next. Second, it provides a place to notice and recover from errors: if an observation returns an unexpected result, the thought step can register the discrepancy and choose a corrective action. ReAct outperforms both chain-of-thought alone (no tools, so it cannot retrieve live facts or run exact arithmetic) and action-only approaches (no reasoning trace, so the model loses track of multi-step plans). It became the standard backbone for almost all subsequent agent frameworks. See [[Chain of Thought]] for the reasoning-only baseline this builds on.

The format is implemented through structured prompting or, in modern systems, through the model's built-in ''function calling'' or ''tool use'' API (available in GPT-4, Claude, and Gemini). In function calling, the model emits a structured JSON tool call rather than free-text, the runtime executes it, and the result is injected back as a special message type. This is more reliable than parsing free-text actions and dramatically reduces format errors.

---

!! What the Math Is Actually Doing

An agent trajectory is a sequence: $(s_1, a_1, o_1,\ s_2, a_2, o_2,\ \ldots,\ s_T, a_T)$ where $s_t$ is the full context at step $t$ (all prior thoughts, actions, and observations concatenated), $a_t$ is the action the model takes (a tool call or a final answer), and $o_t$ is the observation returned by the tool. At each step, the LLM samples:

$$a_t \sim \pi_\theta(a_t \mid s_t)$$

where $\pi_\theta$ is the language model's conditional distribution over next tokens. The context $s_t$ grows monotonically: $s_{t+1} = s_t \oplus a_t \oplus o_t$, where $\oplus$ denotes concatenation. The agent terminates when the model emits a special ''final answer'' action rather than a tool call.

The context window is the agent's working memory. Its finite length (128K–1M tokens in current models) sets a hard ceiling on the number of steps the agent can take before old context must be dropped or summarized. Planning strategies differ in how they structure the generation: ''ReAct'' interleaves thoughts and actions in a flat chain; ''Tree of Thought'' branches the search, generating multiple candidate continuations at each step and using the model to score and prune them; ''Plan-and-Solve'' generates a full plan first, then executes it step by step, reducing mid-task drift.

External memory extends the effective horizon beyond the context window. ''In-context memory'' stores observations directly in the prompt (cheap, limited). ''Retrieval-augmented memory'' (RAG) stores observations in a vector database and retrieves relevant ones at each step (expensive to query, essentially unlimited capacity). ''Episodic memory'' records summaries of prior task runs and retrieves them when facing similar tasks, allowing the agent to improve over time without retraining. See [[RAG]] for the retrieval mechanism in detail.

!! Multi-Agent Systems and Tool Design

Single agents hit two hard limits: the context window fills up during long tasks, and errors compound — a wrong observation at step 3 corrupts the reasoning for all subsequent steps. ''Multi-agent'' architectures address both. A ''planner'' agent decomposes a complex goal into subtasks and delegates each to a ''specialist'' agent with a focused role: a researcher agent does web retrieval, a coder agent writes and executes code, a critic agent checks outputs for errors. The planner synthesizes results. Because each specialist operates over a shorter trajectory, context limits are less binding. Frameworks implementing this pattern include ''LangChain'', ''AutoGen'' (Wu et al. 2023), and ''CrewAI''.

''Tool design'' turns out to matter as much as model capability. A tool with a vague description, no error messages, or unstructured output will fail constantly — the model cannot interpret malformed returns and tends to hallucinate plausible-looking results when observations are ambiguous. A well-designed tool has: a precise name, a one-sentence docstring explaining exactly what it does, typed parameters with examples, and structured JSON output with explicit error fields. Studies have shown that improving tool descriptions can increase agent success rate by 20–30 percentage points with no change to the model.

''Code execution'' is the most powerful single tool. It lets the agent offload all arithmetic to a Python interpreter, manipulate structured data with pandas, and — critically — //verify its own outputs// by running unit tests or assertions. An agent that can write a test and see it fail has a self-correction mechanism that dramatically improves reliability on software engineering tasks. ''Reflexion'' (Shinn et al. 2023) formalizes this: after each failed attempt, the agent writes a verbal self-critique describing what went wrong, stores it in memory, and uses it to guide the next attempt. This simple loop more than doubles success rates on certain benchmarks.

---

!! When LLM Agents Break Down

''Error propagation and cascading failures:'' Because each step's observation becomes part of the context for all subsequent steps, a single incorrect tool call — a search that returns a misleading snippet, a code bug that silently produces the wrong number — corrupts the reasoning chain from that point forward. The model tends to trust its own prior outputs; it rarely spontaneously backtracks to question an earlier observation. On long tasks (20+ steps), even a 5% per-step error rate compounds to near-certain failure at the trajectory level. Self-consistency (sampling multiple trajectories and taking a majority vote) helps for tasks with verifiable answers but is not practical for open-ended tasks where there is no ground-truth to vote on.

''Hallucinated tool calls and confabulation:'' When a model is uncertain or when a task requires knowledge outside its training distribution, it may emit a plausible-looking tool call with fabricated parameters, interpret a vague observation as confirmation, and proceed confidently toward a wrong answer. This is particularly dangerous in agentic settings because the model's confident prose gives no signal that the underlying information is fabricated. Unlike a chatbot where hallucination is visible in the output, an agent's hallucination can be buried in the middle of a 40-step trajectory and go unnoticed until the final answer is checked.

''Long-horizon planning and goal drift:'' Current agents are much better at executing a pre-specified plan than at maintaining coherent long-horizon goals autonomously. Over very long trajectories, models tend to lose track of the original objective, over-commit to an early interpretation of the task, or get stuck in repetitive loops of similar tool calls. Benchmarks like SWE-bench (software engineering) and WebArena (web navigation) currently see success rates of roughly 20–50% on realistic tasks for the best available models, suggesting that reliable long-horizon planning remains an open research problem.

---

!! Technical Details

''Foundational papers:'' Yao et al. 2023 ("ReAct: Synergizing Reasoning and Acting in Language Models") established the interleaved thought-action-observation format. Shinn et al. 2023 ("Reflexion: Language Agents with Verbal Reinforcement Learning") added verbal self-critique and memory. Wang et al. 2023 ("Plan-and-Solve Prompting") showed that planning before acting reduces mid-task drift. Wu et al. 2023 ("AutoGen: Enabling Next-Gen LLM Applications via Multi-Agent Conversation") introduced the multi-agent conversation framework.

''API milestones:'' OpenAI function calling (June 2023) standardized structured tool-use format in GPT-4. Anthropic tool use in Claude (2023) followed. Both allow the model to emit typed JSON function calls rather than free-text actions, and to receive structured observations, reducing parsing failures dramatically.

''Benchmarks:'' ''WebArena'' (Zhou et al. 2023) — 812 realistic web navigation tasks (shopping, Reddit, GitLab) requiring multi-step browser interaction; best agents score ~35–45%. ''SWE-bench'' (Jimenez et al. 2023) — 2,294 real GitHub issues requiring code changes to pass tests; best agents (with scaffolding) score ~20–50% depending on difficulty split. ''HotpotQA'' and ''FEVER'' — multi-hop question answering and fact verification, earlier benchmarks where ReAct showed the first strong results.

''Planning strategies compared:'' ReAct (flat chain, interleaved) is simplest and most widely deployed. Tree of Thought (branching + search) is more powerful on combinatorial tasks but requires $O(b^d)$ model calls for branching factor $b$ and depth $d$. Plan-and-Solve generates a plan in one pass and executes it; faster but brittle if the plan is wrong. Reflexion adds a memory-augmented outer loop: attempt → self-critique → revised attempt; effective but requires multiple episodes per task.

''Current reliability:'' State-of-the-art agents using GPT-4 or Claude 3+ with well-designed tool scaffolding succeed on roughly 20–50% of complex, real-world tasks depending on domain. Key bottlenecks are error recovery, long-horizon coherence, and context-window exhaustion. Active research directions include agent fine-tuning (training models specifically on agent trajectories), hierarchical planning, and formal verification of intermediate steps.
